## 1. 模块概述

### 1.1 模块定位

架构中心是 Mate Platform 的企业 EA（Enterprise Architecture）架构管理中心。提供从业务架构、应用架构、数据架构到技术架构的全栈架构资产沉淀、可视化与治理能力，并通过与 Ontology 本体引擎的深度联动，建立架构元素与企业语义模型的双向映射。

架构中心不是一个独立的架构文档库，而是贯穿业务、应用、数据、技术四层的活态治理框架。它帮助企业回答：当前具备哪些业务能力？某个业务流程变更会影响哪些系统、数据、接口？数据资产的定义来源是业务对象还是技术表？技术平台建设是否对齐了业务战略？

### 1.2 目标用户

| 角色 | 说明 | 典型场景 |
|---|---|---|
| 企业架构师 | 负责整体架构规划与设计 | 绘制业务能力地图、定义应用间依赖关系、规划架构路线图 |
| IT 管理者 | 负责 IT 资产管理与技术决策 | 审查技术债务、评估技术选型、审批架构变更 |
| 业务架构师 | 负责业务能力建模 | 梳理价值流、定义业务能力层级、映射 Ontology 实体 |
| 数据架构师 | 负责数据资产规划 | 定义数据主题域、绘制数据流转图、管理数据资产目录 |
| 架构评审委员 | 参与架构评审决策 | 审阅架构评审申请、出具评审意见、跟踪评审决议落实 |

### 1.3 核心价值

- 全栈贯通：业务-应用-数据-技术四层架构统一管理，跨层关联可视化
- 活态资产：架构资产与 Ontology 联动，变更自动传播，避免文档与实际脱节
- 影响驱动：任意架构元素变更可一键发起影响分析，自动推导受影响范围
- 治理闭环：架构评审流程化，标准规范结构化，技术债务可追踪

### 1.4 核心概念

| 概念 | 说明 | 示例 |
|---|---|---|
| 业务能力（Business Capability） | 企业为达成战略目标所需具备的业务功能，以层级树组织 | "客户管理"→"客户续约管理" |
| 价值流（Value Stream） | 为交付价值给内外部客户而串联的端到端业务活动序列 | "线索到现金"、"采购到付款" |
| 应用系统（Application System） | 支撑业务能力运转的信息系统 | "CRM 系统"、"ERP 系统" |
| 技术栈（Tech Stack） | 应用系统使用的技术组件与技术平台 | "Java 21 + Spring Boot 3.4" |
| 架构资产（EA Asset） | 四层架构中的所有可治理对象 | 业务能力、应用、数据实体、技术组件 |
| 架构评审（Architecture Review） | 对重大架构变更的正式审查与决策流程 | 新建系统技术选型评审 |

- **搜索栏：** 二级 Tab 下方左侧，输入框，placeholder="搜索业务能力名称"，输入后实时过滤
- **筛选区：** 搜索栏右侧，下拉筛选「业务域」（全部/销售域/供应链域/财务域/人力域/IT 域）、「状态」（全部/规划中/已激活/已废弃）
- **操作按钮：** 右上角「创建能力地图」蓝色主按钮
- **视图切换：** 按钮组，切换"树形视图"与"矩阵视图"

**能力地图卡片列表（树形视图）：**

| 元素 | 说明 |
|---|---|
| 能力名称 | 加粗 16px，可折叠展开子能力 |
| 能力编码 | 名称右侧，灰色 12px，格式 `CAP_XXXX` |
| 能力描述 | 编码下方，灰色 14px，最多 2 行 |
| 所属业务域 | 标签颜色：销售域(蓝色) / 供应链域(绿色) / 财务域(橙色) / 人力域(紫色) / IT 域(青色) |

| Ontology 映射状态 | 标签：已映射(绿色) / 未映射(灰色) / 部分映射(黄色) |
| 更新时间 | 卡片底部右侧，格式"2026-07-16 更新" |

**空状态：** 列表为空时，显示居中引导插画 + 文案"还没有业务能力地图，点击创建第一张能力地图" + 「创建能力地图」按钮

**交互：** 点击能力节点 → 跳转到能力详情页（3.1.5）

#### 3.1.2 创建业务能力地图

**入口：** 业务架构总览页右上角「创建能力地图」按钮，或空状态页的「创建能力地图」按钮

**操作步骤：**

1. 用户点击「创建能力地图」按钮 → 系统弹出创建能力地图模态对话框
2. 用户在对话框中填写以下表单：

| 字段 | 类型 | 必填 | 校验规则 | 默认值 | 说明 |
|---|---|---|---|---|---|
| 能力地图名称 | 文本输入 | 是 | 1-50 字符，不能纯空格 | - | 如"销售管理能力地图"、"供应链能力地图" |
| 能力地图编码 | 文本输入 | 是 | 字母+数字+下划线，3-30 字符，全局唯一 | - | 创建后不可修改，作为系统标识 |
| 所属业务域 | 下拉选择 | 是 | 选择已有业务域或输入新业务域名 | - | 销售域/供应链域/财务域/人力域/IT 域/自定义 |
| 能力地图描述 | 多行文本 | 否 | 最多 500 字符 | - | 描述能力地图覆盖范围与目标 |
| 根能力名称 | 文本输入 | 是 | 1-30 字符 | - | 能力地图的根节点名称，如"销售管理" |
| 版本 | 文本输入 | 否 | 语义化版本号 | v1.0 | 能力地图版本标识 |

3. 用户点击「保存」按钮 → 系统执行以下校验：
   - 能力地图名称：非空校验 → 失败则输入框下方红字提示"能力地图名称不能为空"
   - 能力地图编码：非空 + 格式 + 唯一性校验 → 失败则提示"能力地图编码只能包含字母、数字和下划线"或"能力地图编码已存在"
   - 所属业务域：非空校验 → 失败提示"请选择所属业务域"
   - 根能力名称：非空校验 → 失败提示"根能力名称不能为空"
   - 校验通过 → 调用后端 API 创建能力地图
4. 创建成功 → 关闭对话框 + 顶部 Toast 绿色提示"能力地图创建成功" + 列表自动刷新，新能力地图出现在列表第一个
5. 创建失败（网络异常等）→ 顶部 Toast 红色提示"创建失败，请重试" + 对话框保持打开

**按钮：**
- 「取消」灰色按钮：点击关闭对话框，不保存任何内容
- 「保存」蓝色主按钮：点击触发校验和保存

#### 3.1.3 添加业务能力节点

**入口：** 能力地图详情页（3.1.5）→ 点击工具栏「添加能力」按钮，或右键点击能力节点 → 选择「添加子能力」

**操作步骤：**

1. 用户点击「添加能力」按钮 → 系统弹出添加能力模态对话框
2. 用户在对话框中填写以下表单：

| 字段 | 类型 | 必填 | 校验规则 | 默认值 | 说明 |
|---|---|---|---|---|---|
| 能力名称 | 文本输入 | 是 | 1-50 字符，不能纯空格，同一父节点下不重名 | - | 如"客户管理"、"合同管理" |
| 能力编码 | 文本输入 | 是 | 字母+数字+下划线，3-30 字符，能力地图内唯一 | - | 创建后不可修改 |
| 父级能力 | 树形下拉 | 是 | 选择父能力节点，默认为当前选中节点或根节点 | 当前选中节点 | 决定能力在层级树中的位置 |
| 能力描述 | 多行文本 | 否 | 最多 500 字符 | - | 描述该能力的业务含义与边界 |
| 能力级别 | 下拉选择 | 否 | L1/L2/L3/L4 | 根据父级自动推断 | 能力层级深度标识 |
| 责任组织 | 部门选择器 | 否 | 从组织架构中选择 | - | 该能力的主要负责部门 |
| 责任人 | 人员选择器 | 否 | 从组织人员中选择 | - | 该能力的主要负责人 |
| 能力状态 | 单选按钮组 | 是 | 必选 | 规划中 | 规划中/已激活/已废弃 |
| 关键性 | 单选按钮组 | 否 | - | 普通 | 核心/重要/普通 |

3. 用户点击「保存」→ 系统校验：
   - 能力名称非空 → 失败提示"能力名称不能为空"
   - 能力编码非空 + 格式 + 地图内唯一 → 失败提示"能力编码已存在"或"能力编码只能包含字母、数字和下划线"
   - 父级能力必须选择 → 失败提示"请选择父级能力"
   - 校验通过 → 在能力地图中新增节点 + Toast"能力节点添加成功" + 能力地图画布刷新，新节点出现在父节点下方
4. 用户点击「取消」→ 关闭对话框

**画布反馈：**
- 新节点以圆角矩形显示，默认蓝色边框
- 父子关系以连线表示，自动布局
- 新增节点自动高亮 2 秒后恢复正常

#### 3.1.4 建立能力层级与调整

**入口：** 能力地图详情页 → 在画布中拖拽能力节点，或右键点击能力节点

**操作步骤：**

1. **拖拽调整层级：** 用户在画布中长按某能力节点 → 拖拽到另一个节点上方释放 → 系统弹出确认对话框"将「{能力A}」移动为「{能力B}」的子能力？" → 点击「确认」→ 层级关系更新 + 画布自动重新布局 + Toast"能力层级已调整"
2. **右键菜单操作：** 右键点击能力节点 → 弹出上下文菜单：
   - 「添加子能力」→ 跳转到 3.1.3 添加能力对话框
   - 「编辑能力」→ 跳转到 3.1.6 编辑能力
   - 「删除能力」→ 跳转到删除确认
   - 「关联应用」→ 弹出应用选择器，关联支撑该能力的应用系统
   - 「关联 Ontology」→ 跳转到 3.6.2 Ontology 映射
   - 「查看影响分析」→ 跳转到 3.5.3 影响分析
3. **删除能力：** 用户点击「删除能力」→ 弹出确认对话框"确定要删除能力「{能力名称}」吗？其所有子能力将一并删除，关联的应用映射和 Ontology 映射也将解除。" → 用户输入能力名称确认 → 点击「确认删除」红色按钮 → 删除成功 + Toast"能力已删除" + 画布刷新

| 属性映射 | 键值对配置 | 业务能力属性 → Ontology 属性的对应关系 |
| 映射说明 | 多行文本 | 映射关系的补充说明 |

5. 用户点击「保存映射」→ 系统校验：
   - 必须选择 Ontology 概念 → 失败提示"请选择 Ontology 概念"
   - 校验通过 → 调用后端保存映射关系 → Toast"映射关系已保存" + 列表刷新，映射状态更新为"已映射"
6. 映射保存后：
   - 能力地图画布中该能力节点的 Ontology 映射标识变为绿色勾
   - Ontology 中的对应 Concept 自动标注"被业务能力「{名称}」引用"
   - 影响分析（3.5.3）中可追溯到此映射关系

**解除映射操作步骤：**
1. 用户点击「解除映射」→ 弹出确认对话框
2. 文案"确定解除业务能力「{能力名}」与 Ontology 概念「{概念名}」的映射关系？解除后影响分析将无法追踪此关联。"
3. 用户点击「确认解除」→ 映射关系删除 → Toast"映射已解除" + 列表刷新
4. 用户点击「取消」→ 关闭对话框

**批量映射操作步骤：**
1. 用户在列表中勾选多个未映射的业务能力 → 点击「批量映射」按钮
2. 系统弹出批量映射面板
3. 对每个选中的能力，用户选择对应的 Ontology 概念
4. 用户点击「批量保存」→ 系统批量保存 → Toast"已批量映射 {N} 个业务能力" + 列表刷新

#### 3.6.3 数据实体→Ontology 实体映射

**入口：** Ontology 联动总览页 → 二级 Tab 选择「数据实体映射」

**页面路径：** `/arch/ontology-mapping/entities`

**页面布局：**

- **搜索栏：** 搜索数据实体名称
- **筛选区：** 筛选「映射状态」、「所属主题域」
- **列表区域：** 表格布局

**映射列表：**

| 列 | 说明 |
|---|---|
| 数据实体 | 实体名称 + 编码 |
| 所属主题域 | 标签 |
| 数据类型 | 业务实体/主数据/参考数据 |
| 来源系统 | 应用名称 |
| Ontology 实体 | 已映射的 Ontology Entity 名称 |
| 字段映射率 | 如"8/10 字段已映射" |
| 映射状态 | 已映射 / 未映射 / 部分映射 |
| 操作 | 配置映射 / 查看详情 / 解除映射 |

**配置映射操作步骤：**

1. 用户点击「配置映射」→ 弹出映射配置对话框
2. 对话框显示：
   - 左侧：数据实体的字段列表（字段名称、数据类型、描述）
   - 右侧：Ontology Entity 选择器 + 属性列表
3. 用户在右侧搜索并选择对应的 Ontology Entity（如数据实体"客户"映射到 Ontology Entity "Customer"）
4. 选择 Entity 后，系统自动加载 Ontology Entity 的属性列表
5. 用户配置字段级映射：
   - 数据实体字段列表与 Ontology 属性列表并排展示
   - 用户通过下拉选择，将每个数据实体字段映射到对应的 Ontology 属性
   - 可拖拽连线建立映射关系
   - 未映射的字段以灰色标识

| 配置项 | 说明 |
|---|---|
| Ontology Entity | 搜索下拉选择对应的 Entity |
| 字段映射 | 数据实体字段 → Ontology 属性的逐字段映射 |
| 映射类型 | 完全等价 / 字段子集 / 字段超集 |
| 数据同步方向 | 单向（架构→Ontology）/ 双向（架构↔Ontology） |
| 映射说明 | 多行文本补充说明 |

6. 用户点击「保存映射」→ 校验：
   - 必须选择 Ontology Entity → 失败提示"请选择 Ontology 实体"
   - 至少映射 1 个字段 → 失败提示"请至少映射一个字段"
   - 校验通过 → 保存 → Toast"数据实体映射已保存" + 列表刷新
7. 映射保存后：
   - 数据资产目录中该实体的 Ontology 映射标签变为"已映射"
   - Ontology 中对应 Entity 自动标注数据来源
   - 数据流转图中该实体节点显示映射标识

#### 3.6.4 映射一致性检查

**入口：** Ontology 联动总览页 → 二级 Tab 选择「一致性检查」

**页面路径：** `/arch/ontology-mapping/consistency`

**页面布局：**

- **操作栏：** 「执行一致性检查」蓝色按钮 + 上次检查时间
- **检查结果摘要：** 总映射数、一致数、不一致数、缺失映射数
- **检查结果列表：**

| 列 | 说明 |
|---|---|
| 检查项 | 如"业务能力「客户管理」→Ontology 概念「Customer」" |
| 检查类型 | 映射存在性/属性一致性/引用有效性 |
| 检查结果 | 一致(绿色) / 不一致(红色) / 警告(黄色) |
| 问题描述 | 如"Ontology 概念「Customer」已被重命名，映射需更新" |
| 建议 | 如"请重新选择对应的 Ontology 概念" |
| 操作 | 修复映射 / 忽略 |

**执行一致性检查操作步骤：**

1. 用户点击「执行一致性检查」→ 系统弹出确认"将检查所有架构资产与 Ontology 的映射一致性，预计需要 30 秒。"
2. 用户点击「确认」→ 系统执行检查（调用 TECH-EA + TECH-ONT 联合校验）
3. 检查过程中显示进度条
4. 检查完成 → 结果列表刷新 → Toast"一致性检查完成，发现 {N} 个问题"
5. 用户可针对每个问题：
   - 点击「修复映射」→ 跳转到对应的映射配置页面（3.6.2 或 3.6.3）
   - 点击「忽略」→ 标记为已忽略，不再提示

**检查规则说明：**

| 检查类型 | 检查内容 | 触发条件 |
|---|---|---|
| 映射存在性 | 架构资产是否已建立 Ontology 映射 | 新增架构资产后未配置映射 |
| 属性一致性 | 映射的字段是否与 Ontology 属性匹配 | Ontology 属性变更后字段映射失效 |
| 引用有效性 | 映射的 Ontology 概念/实体是否仍然存在 | Ontology 概念/实体被删除或重命名 |
| 双向一致 | Ontology 侧是否标注了对应的架构资产 | 架构资产已映射但 Ontology 侧未反向标注 |

```
CapabilityMap（业务能力地图）
├── id: string (UUID)
├── name: string
├── code: string (unique)
├── businessDomain: string
├── description: string
├── version: string
├── status: enum (draft / active / archived)
├── createdBy: string
├── createdAt: datetime
├── updatedAt: datetime
└── capabilities: BusinessCapability[]

BusinessCapability（业务能力）
├── id: string (UUID)
├── mapId: string (FK → CapabilityMap)
├── parentId: string (FK → BusinessCapability, nullable)
├── name: string
├── code: string (unique within map)
├── description: string
├── level: string (L1/L2/L3/L4/L5/L6)
├── status: enum (planning / active / deprecated)
├── criticality: enum (core / important / normal)
├── responsibleOrg: string
├── responsiblePerson: string
├── ontologyConceptId: string (nullable, FK → Ontology Concept)
├── createdAt: datetime
└── updatedAt: datetime

ValueStream（价值流）
├── id: string (UUID)
├── name: string
├── code: string (unique)
├── description: string
├── triggerEvent: string
├── endEvent: string
├── status: enum (planning / active / deprecated)
├── stages: ValueStreamStage[]
├── createdAt: datetime
└── updatedAt: datetime

ValueStreamStage（价值流阶段）
├── id: string (UUID)
├── valueStreamId: string (FK)
├── name: string
├── description: string
├── order: int
├── output: string
├── capabilityIds: string[] (FK → BusinessCapability)
└── roleIds: string[]

BusinessProcess（业务流程）
├── id: string (UUID)
├── name: string
├── code: string (unique)
├── valueStreamId: string (FK, nullable)
├── description: string
├── processType: enum (main / sub / auxiliary)
├── frequency: enum (high / medium / low / on_demand)
├── systemIds: string[] (FK → ApplicationSystem)
├── steps: ProcessStep[]
├── createdAt: datetime
└── updatedAt: datetime

ProcessStep（流程步骤）
�├── id: string (UUID)
├── processId: string (FK)
├── name: string
├── order: int
├── roleId: string (FK → BusinessRole)
├── input: string
├── output: string
├── capabilityId: string (FK, nullable)
└── systemId: string (FK, nullable)

BusinessRole（业务角色）
├── id: string (UUID)
├── name: string
├── code: string (unique)
├── businessDomain: string
├── description: string
├── iamRoleId: string (nullable, FK → TECH-IAM Role)
├── createdAt: datetime
└── updatedAt: datetime

ApplicationSystem（应用系统）
├── id: string (UUID)
├── name: string
├── code: string (unique)
├── description: string
├── category: enum (core / support / tool)
├── lifecycleStatus: enum (planning / building / running / retiring / retired)
├── responsibleTeam: string
├── responsiblePerson: string
├── goLiveDate: date
├── plannedRetireDate: date
├── vendor: string
├── version: string
├── entryUrl: string
├── healthScore: int (0-100)
├── createdAt: datetime
└── updatedAt: datetime

AppCapabilityRelation（应用-能力关联）
├── id: string (UUID)
├── appId: string (FK → ApplicationSystem)
├── capabilityId: string (FK → BusinessCapability)
├── relationType: enum (primary / secondary / planned)
└── createdAt: datetime

AppDependency（应用依赖关系）
├── id: string (UUID)
├── sourceAppId: string (FK → ApplicationSystem)
├── targetAppId: string (FK → ApplicationSystem)
├── dependencyType: enum (data / service / message / file / api)
├── direction: enum (one_way / bidirectional)
├── strength: enum (strong / medium / weak)
├── interface: string
├── syncMode: enum (sync / async)
├── description: string
├── createdAt: datetime
└── updatedAt: datetime

TechDebt（技术债务）
├── id: string (UUID)
├── appId: string (FK → ApplicationSystem)
├── title: string
├── type: enum (obsolete / architecture_defect / security / performance / maintainability / missing_docs)
├── priority: enum (P0 / P1 / P2 / P3)
├── status: enum (open / in_progress / resolved / ignored)
├── description: string
├── impactScope: string
├── suggestedSolution: string
├── plannedResolveDate: date
├── assignee: string
├── attachments: string[]
├── createdBy: string
├── createdAt: datetime
└── updatedAt: datetime

DataDomain（数据主题域）
├── id: string (UUID)
├── parentId: string (FK, nullable)
├── name: string
├── code: string (unique)
├── description: string
├── dataOwner: string
├── businessOwner: string
├── createdAt: datetime
└── updatedAt: datetime

DataEntity（数据实体）
├── id: string (UUID)
├── domainId: string (FK → DataDomain)
├── name: string
├── code: string (unique)
├── description: string
├── entityType: enum (business / technical / master / reference)
├── sourceSystemId: string (FK → ApplicationSystem, nullable)
├── storageLocation: string
├── updateFrequency: enum (realtime / hourly / daily / weekly / monthly)
├── ontologyEntityId: string (nullable)
├── fields: DataField[]
├── createdAt: datetime
└── updatedAt: datetime

DataField（数据字段）
├── id: string (UUID)
├── entityId: string (FK → DataEntity)
├── name: string
├── code: string
├── dataType: string
├── isPrimaryKey: boolean
├── isNullable: boolean
├── defaultValue: string
├── description: string
└── ontologyAttributeId: string (nullable)

DataFlowDiagram（数据流转图）
├── id: string (UUID)
├── name: string
├── code: string (unique)
├── domainId: string (FK, nullable)
├── description: string
├── nodes: FlowNode[]
├── edges: FlowEdge[]
├── createdAt: datetime
└── updatedAt: datetime

DataAsset（数据资产）
├── id: string (UUID)
├── name: string
├── code: string (unique)
├── domainId: string (FK → DataDomain)
├── dataType: enum (business / master / reference / metric)
├── description: string
├── sourceSystemId: string (FK, nullable)
├── storageLocation: string
├── dataSize: string
├── updateFrequency: enum
├── dataOwner: string
├── securityLevel: enum (public / internal / confidential / top_secret)
├── ontologyEntityId: string (nullable)
├── qualityScore: int (0-100)
├── createdAt: datetime
└── updatedAt: datetime

TechComponent（技术组件）
├── id: string (UUID)
├── name: string
├── code: string (unique)
├── category: enum (language / framework / database / middleware / cloud / devops / monitoring / security)
├── currentVersion: string
├── standardVersion: string
├── status: enum (adopted / evaluating / restricted / deprecated)
├── description: string
├── vendorType: enum (open_source / commercial / hybrid)
├── docUrl: string
├── owner: string
├── usageGuide: string
├── restrictionNote: string
├── createdAt: datetime
└── updatedAt: datetime

InfrastructureTopology（基础设施拓扑图）
├── id: string (UUID)
├── name: string
├── code: string (unique)
├── environment: enum (dev / test / staging / production)
├── description: string
├── nodes: TopologyNode[]
├── edges: TopologyEdge[]
├── createdAt: datetime
└── updatedAt: datetime

TechStandard（技术标准）
├── id: string (UUID)
├── name: string
├── code: string (unique)
├── category: enum (coding / architecture / security / deployment / data)
├── version: string
├── content: text (rich text)
├── status: enum (active / draft / revoked)
├── effectiveDate: date
├── owner: string
├── scope: string
├── attachments: string[]
├── createdAt: datetime
└── updatedAt: datetime

ArchitectureReview（架构评审）
├── id: string (UUID)
├── title: string
├── reviewType: enum (new_system / major_change / tech_selection / retirement)
├── appId: string (FK, nullable)
├── capabilityId: string (FK, nullable)
├── applicant: string
├── applicationDate: date
├── plannedReviewDate: date
├── reviewers: string[]
├── description: string
├── changeSummary: text
├── changeTypes: string[]
├── impactScope: text
├── beforeArchitecture: text
├── afterArchitecture: text
├── techSolution: text
├── riskAssessment: text
├── plannedCompleteDate: date
├── attachments: string[]
├── status: enum (draft / submitted / reviewing / approved / rejected / withdrawn)
├── opinions: ReviewOpinion[]
├── createdAt: datetime
└── updatedAt: datetime

ReviewOpinion（评审意见）
├── id: string (UUID)
├── reviewId: string (FK → ArchitectureReview)
├── reviewer: string
├── opinion: text
├── conclusion: enum (approve / conditional_approve / reject / need_more_info)
├── conditions: text
├── submittedAt: datetime

OntologyMapping（Ontology 映射）
├── id: string (UUID)
├── assetType: enum (capability / data_entity / application)
├── assetId: string
├── ontologyType: enum (concept / entity / action)
├── ontologyId: string
├── mappingType: enum (exact / contains / reference)
├── fieldMappings: json (字段级映射配置)
├── syncDirection: enum (one_way / bidirectional)
├── description: string
├── createdBy: string
├── createdAt: datetime
└── updatedAt: datetime
```
