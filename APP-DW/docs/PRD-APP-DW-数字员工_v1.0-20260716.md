## 1. 模块概述

### 1.1 模块定位

APP-DW 是 Mate Platform 的数字员工管理与应用模块，为企业提供 AI 驱动的自动化劳动力。数字员工能够理解企业制度与流程、执行业务任务、与人类协作、与其他 Agent 交互。每个数字员工具备角色定义、专业知识、操作技能和协作能力，可 7x24 小时不间断工作。

数字员工区别于 APP-SUPERAI（通用超级 AI 助手）的定位：APP-SUPERAI 是面向所有用户的通用对话助手，而 APP-DW 中的数字员工是**岗位化的、有明确角色和职责边界**的专职 AI 员工，每个员工绑定特定知识库、拥有特定技能权限、执行特定类型任务。

### 1.2 核心价值

- **知识提炼**：自动从企业制度、流程文档、访谈记录中提炼 Ontology 知识结构，将非结构化文档转化为可推理的结构化知识
- **自主执行**：基于角色定义自主接受并执行任务，支持复杂多步操作，通过 TECH-AGENT 框架编排推理链路
- **人机协作**：数字员工与人类员工协同工作，能力互补，管理者可实时监控和干预
- **多员工协作**：多个数字员工之间通过 A2A 协议协作完成复杂任务，支持任务委派和结果汇总
- **效果可度量**：对话质量评估、任务完成率、效率提升等量化指标，持续优化数字员工能力

### 1.3 目标用户

| 角色 | 说明 | 典型场景 |
|---|---|---|
| 数字员工管理者 | 负责创建、配置、监控数字员工，评估工作效果 | 创建财务助手数字员工、配置报销审批能力、查看月度效果报表 |
| 业务团队负责人 | 向数字员工分配任务，查看执行结果 | 给数据分析师数字员工分配季度报告任务、查看执行轨迹 |
| 知识工程师 | 管理数字员工的知识库、优化对话策略 | 上传报销制度文档提炼知识、审核 AI 抽取的 Ontology、调整知识范围 |
| 普通用户 | 与数字员工交互、委托任务 | 通过对话界面委托数字员工查询数据、执行操作 |

### 1.4 核心概念

| 概念 | 说明 |
|---|---|
| 数字员工（Digital Employee） | 一个具备角色定义、知识、技能和权限的 AI Agent 实例，可执行业务任务 |
| 角色（Role） | 数字员工的身份定位，如"财务助手""HR助手""数据分析师"，决定其 System Prompt 和能力范围 |
| 能力（Capability） | 数字员工可以执行的操作，包括 Tools（工具调用）、Actions（业务操作）、RAG 知识检索 |
| 知识范围（Knowledge Scope） | 数字员工可访问的知识库集合，决定其专业领域边界 |
| 任务（Task） | 分配给数字员工的工作单元，包含输入参数、执行步骤、输出结果 |
| 协作（Collaboration） | 多个数字员工协同完成复杂任务的编排流程 |
| 执行轨迹（Trace） | 数字员工执行任务时的完整推理链路和操作步骤记录 |

## 3. 功能详情

### 3.1 数字员工管理

#### 3.1.1 员工列表页

**入口：** 点击左侧导航栏「数字员工」菜单，或直接访问 `/dw`

**页面路径：** `/dw`

**页面布局：**

- **顶部统计栏：** 4 个统计卡片横向排列
  - 在线员工数：当前 ACTIVE 状态的数字员工数量
  - 今日任务数：今日创建的任务总数
  - 任务完成率：今日已完成任务 / 今日总任务
  - 平均质量评分：今日所有对话的平均质量分

- **搜索栏：** 统计栏下方左侧，输入框，placeholder="搜索员工名称或角色"，输入后实时过滤列表

- **筛选区：** 搜索栏右侧，下拉筛选
  - 「角色分类」：全部 / 财务类 / HR类 / 法务类 / 数据分析类 / 客服类 / 自定义
  - 「状态」：全部 / 草稿 / 在线 / 已停用 / 已归档

- **操作按钮：** 右上角
  - 「创建数字员工」蓝色主按钮
  - 「员工模板」灰色按钮 → 跳转模板库

- **列表区域：** 卡片网格布局，每行 3 个卡片（响应式）

**员工卡片内容：**

| 元素 | 说明 |
|---|---|
| 头像 | 左上角，64x64 圆形头像，未设置时显示角色首字母 |
| 员工名称 | 头像右侧，加粗 16px |
| 角色标签 | 名称右侧，彩色标签（财务类-蓝色 / HR类-绿色 / 法务类-紫色 / 数据分析类-青色 / 客服类-橙色） |
| 职责描述 | 名称下方，灰色 14px，最多显示 2 行 |
| 能力图标 | 卡片中部，小图标行：显示已配置的能力类型（Tool图标 / RAG图标 / Action图标 / 模型图标），hover显示数量 |
| 今日任务数 | 卡片底部左侧，格式"今日 5 个任务" |
| 质量评分 | 卡片底部中部，星级显示（1-5星），格式"4.3 / 5.0" |
| 状态标签 | 卡片右上角：草稿(灰色) / 在线(绿色脉冲动画) / 已停用(灰色) / 已归档(暗灰色) |
| 更新时间 | 卡片底部右侧，格式"2026-07-16 更新" |

**卡片操作菜单：** 卡片右上角「...」按钮 → 下拉菜单
- 查看详情 → 跳转员工详情页（3.1.7）
- 编辑配置 → 跳转编辑页（3.1.3）
- 启用/停用 → 切换状态（3.1.4）
- 克隆 → 克隆员工（3.1.6）
- 归档 → 归档员工
- 删除 → 删除员工（3.1.5）

**空状态：** 列表为空时，显示居中引导插画 + 文案"还没有数字员工，点击创建第一位吧" + 「创建数字员工」按钮

**交互：** 点击员工卡片主体 → 跳转到员工详情页（3.1.7）

**第一步：填写基本信息**

**操作步骤：**

1. 用户进入创建向导，默认显示第一步「基本信息」，步骤导航条中①高亮
2. 用户填写以下表单：

| 字段 | 类型 | 必填 | 校验规则 | 默认值 | 说明 |
|---|---|---|---|---|---|
| 员工名称 | 文本输入 | 是 | 2-30 字符，不能纯空格，全局唯一 | - | 数字员工的显示名称，如"财务小助手" |
| 员工编码 | 文本输入 | 是 | 字母+数字+下划线，2-30 字符，全局唯一 | 自动生成（如 DW_001） | 系统标识，创建后不可修改 |
| 角色分类 | 下拉选择 | 是 | 必选 | - | 财务类 / HR类 / 法务类 / 数据分析类 / 客服类 / 自定义 |
| 角色身份 | 文本输入 | 是 | 1-50 字符 | - | 如"报销审批助手""薪酬查询专员"，根据角色分类自动推荐 |
| 头像 | 图片上传 | 否 | JPG/PNG，<2MB | 角色分类默认头像 | 点击上传或选择预设头像 |
| 职责描述 | 多行文本 | 是 | 10-500 字符 | - | 描述该数字员工的职责范围和工作目标 |
| System Prompt | 多行文本 | 否 | 最多 2000 字符 | 根据角色自动生成 | 数字员工的系统提示词，高级用户可自定义 |

3. **角色模板快捷选择：** 表单上方显示一排预设角色模板卡片（如"财务助手""HR助手""数据分析师""法务顾问""客服专员"），点击模板卡片 → 自动填充角色分类、角色身份、职责描述和 System Prompt，用户可在填充基础上修改
4. 填写完成后，用户点击底部「下一步」按钮
5. 系统执行校验：
   - 员工名称非空 + 唯一性校验 → 失败提示"员工名称不能为空"或"该名称已存在，请更换"
   - 员工编码非空 + 格式 + 唯一性校验 → 失败提示"员工编码只能包含字母、数字和下划线"或"员工编码已存在"
   - 角色分类必选 → 失败提示"请选择角色分类"
   - 职责描述不少于 10 字符 → 失败提示"职责描述至少 10 个字符"
6. 校验通过 → 步骤导航条②高亮，进入第二步

**反馈：** 校验失败时对应字段下方红字提示；校验通过平滑切换到下一步

**第二步：配置能力**

**操作步骤：**

1. 进入第二步「能力配置」，页面分 4 个配置区域，以折叠面板形式展示：

**区域 A - Tool 工具选择（3.2.1 详述）**

| 配置项 | 说明 |
|---|---|
| 可用工具列表 | 从 TECH-MCP 注册的 Tools 中选择，勾选式 |
| 搜索框 | 按工具名称搜索 |
| 工具分类筛选 | 全部 / 数据查询 / 文件操作 / 通知推送 / 外部API |
| 已选工具 | 右侧显示已勾选的工具列表，可拖拽排序（决定调用优先级） |

操作：用户在左侧工具列表中勾选需要的工具 → 右侧"已选工具"区域实时更新 → 点击工具可展开参数配置（如默认参数值、超时时间）

**区域 B - Action 权限配置（3.2.2 详述）**

| 配置项 | 说明 |
|---|---|
| 可用 Action 列表 | 从 TECH-ACTION 注册的 Action 中选择 |
| Action 权限级别 | 每个可选：仅查询 / 查询+执行 / 完全控制 |
| 执行确认策略 | 自动执行 / 需人工确认 / 高风险需审批 |

操作：用户勾选 Action → 设置权限级别 → 设置执行确认策略

**区域 C - RAG 知识库绑定（3.2.3 详述，详细在第三步设定范围）**

| 配置项 | 说明 |
|---|---|
| 绑定知识库 | 从 TECH-RAG 已创建的知识库列表中选择 |
| 检索策略 | 混合检索（向量+关键词） / 纯向量检索 / 纯关键词检索 |
| Top-K 值 | 检索返回的文档数量，默认 5 |
| 重排序 | 开启 / 关闭（开启后使用 reranker 模型对检索结果重排序） |

操作：用户勾选知识库 → 配置检索参数

**区域 D - 模型选择（3.2.4 详述）**

| 配置项 | 说明 |
|---|---|
| LLM 模型 | 从 TECH-LLMGW 可用模型列表中选择，下拉显示模型名称+能力说明 |
| Temperature | 滑块，0.0-1.0，默认 0.7 |
| Max Tokens | 数字输入，100-8192，默认 4096 |
| 对话风格 | 单选：严谨（低温度）/ 均衡（中温度）/ 灵活（高温度） |

操作：用户选择模型 → 调整参数或选择对话风格（选择风格自动设置温度）

2. 配置完成后，用户点击底部「下一步」按钮
3. 系统校验：
   - 至少配置 1 项能力（Tool 或 Action 或 RAG 知识库）→ 失败提示"请至少配置一项能力"
   - 已选模型在 TECH-LLMGW 中可用 → 失败提示"所选模型不可用，请重新选择"
4. 校验通过 → 步骤导航条③高亮，进入第三步

---

**第三步：设定知识范围**

**操作步骤：**

1. 进入第三步「知识范围」，页面显示两部分：

**Part A - 知识库范围确认**

显示第二步已绑定的知识库列表，此处可进一步细化：

| 配置项 | 类型 | 说明 |
|---|---|---|
| 知识库列表 | 表格 | 显示已绑定知识库名称、文档数、更新时间 |
| 知识过滤条件 | 条件构建器 | 可设置仅检索特定分类、特定时间段的文档 |
| 权限范围说明 | 多行文本 | 描述该数字员工的知识边界，写入 System Prompt |

操作：
- 用户可点击「添加知识库」从 TECH-RAG 知识库列表中追加
- 用户可点击知识库行右侧「移除」按钮取消绑定
- 用户可点击「设置过滤条件」弹出条件构建器

**Part B - 知识提炼快捷入口**

| 元素 | 说明 |
|---|---|
| 提示卡片 | "还没有足够的知识？上传企业制度/流程文档，AI 自动提炼知识结构" |
| 「上传文档提炼」按钮 | 点击 → 跳转到知识提炼页面（3.3），提炼完成后回到此处 |

操作：用户可点击「上传文档提炼」跳转到知识提炼流程，提炼完成自动返回

2. 配置完成后，用户点击底部「下一步」按钮
3. 系统校验：
   - 如未绑定任何知识库，弹出确认框"该数字员工未绑定知识库，可能影响回答准确性。确定继续？"→ 用户可选择「继续」或「返回配置」
4. 校验通过 → 步骤导航条④高亮，进入第四步

---

**第四步：确认并完成创建**

**操作步骤：**

1. 进入第四步「确认创建」，页面显示前 3 步配置的汇总信息：

**汇总卡片：**

| 区域 | 显示内容 |
|---|---|
| 基本信息 | 员工名称、编码、角色分类、角色身份、头像预览、职责描述 |
| 能力配置 | 已选 Tools 列表、已选 Actions 列表（含权限级别）、模型名称+参数 |
| 知识范围 | 已绑定知识库列表、检索策略、Top-K 值 |
| 对话策略 | 对话风格、System Prompt 预览 |

2. 用户可点击各区域右上角「编辑」按钮 → 跳回对应步骤修改
3. 用户确认无误后，点击底部「完成创建」蓝色主按钮
4. 系统执行创建：
   - 调用 `POST /api/v1/dw/employees` 创建数字员工
   - 创建过程中显示 Loading 动画"正在创建数字员工..."
5. 创建成功 → Toast 绿色提示"数字员工「{员工名称}」创建成功" → 自动跳转到员工详情页（3.1.7）
6. 创建失败 → Toast 红色提示"创建失败：{错误原因}" → 停留在当前页面

**底部按钮说明：**
- 「上一步」灰色按钮：返回上一步
- 「保存为草稿」灰色按钮：保存当前配置为草稿状态，可稍后继续编辑
- 「完成创建」蓝色主按钮：提交并创建

#### 3.1.3 编辑配置

**入口：** 员工列表页卡片操作菜单「编辑配置」，或员工详情页右上角「编辑」按钮

**页面路径：** `/dw/employees/{employeeId}/edit`

**操作步骤：**

1. 用户点击「编辑配置」→ 进入编辑页面，页面布局与创建向导（3.1.2）一致，4 步导航条全部可点击跳转
2. 系统加载当前数字员工的配置数据，预填到各步骤表单中
3. 员工编码字段显示为灰色只读状态（创建后不可修改）
4. 用户可自由跳转到任意步骤修改配置
5. 用户修改完成后，点击底部「保存修改」按钮
6. 系统校验（同创建向导校验规则，跳过编码唯一性校验）
7. 校验通过 → 调用 `PUT /api/v1/dw/employees/{employeeId}` 更新配置
8. 保存成功 → Toast 提示"配置已更新" → 跳转回员工详情页
9. 如果数字员工处于 ACTIVE 状态，修改配置后弹出提示"配置已更新，需要重启数字员工才能生效。是否立即重启？"→「立即重启」/「稍后重启」

**版本管理：** 每次保存修改自动生成新版本，可在员工详情页查看版本历史和回滚

#### 3.1.4 启用/停用数字员工

**入口：** 员工列表页卡片操作菜单「启用」/「停用」，或员工详情页右上角状态切换开关

**操作步骤 - 停用：**

1. 用户点击「停用」→ 系统弹出确认对话框
2. 对话框内容：黄色警告图标 + 文案"停用「{员工名称}」后，该员工将无法接受新任务和响应对话。进行中的任务将继续完成。确定停用？"
3. 用户点击「确认停用」→ 系统调用 `POST /api/v1/dw/employees/{employeeId}/deactivate`
4. 停用成功 → Toast 提示"数字员工已停用" → 员工状态标签变为灰色"已停用"
5. 用户点击「取消」→ 关闭对话框

**操作步骤 - 启用：**

1. 用户点击「启用」→ 系统调用 `POST /api/v1/dw/employees/{employeeId}/activate`
2. 启用成功 → Toast 提示"数字员工已上线" → 员工状态标签变为绿色"在线"（带脉冲动画）
3. 如果配置不完整（如绑定的知识库已删除），提示"启用失败：知识库「{名称}」已不存在，请重新配置"

#### 3.1.5 删除数字员工

**入口：** 员工列表页卡片操作菜单「删除」，或员工详情页右上角「更多操作」→「删除」

**操作步骤：**

1. 用户点击「删除」→ 系统弹出确认对话框
2. 对话框内容：红色警告图标 + 文案"确定要删除数字员工「{员工名称}」吗？删除后所有配置、对话记录和任务历史将永久删除，无法恢复。"
3. 用户需输入员工名称以确认（二次确认，防止误删）
4. 用户输入正确的员工名称后，「确认删除」红色按钮变为可点击状态
5. 用户点击「确认删除」→ 系统调用 `DELETE /api/v1/dw/employees/{employeeId}`
6. 删除成功 → Toast 提示"数字员工已删除" → 返回员工列表页，列表自动刷新

**约束：**
- 在线状态（ACTIVE）的数字员工不能直接删除，需先停用
- 有进行中任务的数字员工不能删除，需等待任务完成或取消
- 删除操作不可撤销

#### 3.1.6 克隆数字员工

### 3.2 能力配置

#### 3.2.1 Tool 工具选择

**入口：** 创建/编辑向导 → 第二步「能力配置」→ 区域 A「Tool 工具选择」

**操作步骤：**

1. 用户展开「Tool 工具选择」折叠面板 → 显示工具选择区域，左右分栏布局
2. **左侧 - 可用工具列表：**
   - 顶部搜索框：placeholder="搜索工具名称"，输入实时过滤
   - 分类筛选 Tab：全部 / 数据查询 / 文件操作 / 通知推送 / 外部API / 自定义
   - 工具列表：每行显示工具名称、图标、分类标签、简短描述

3. **右侧 - 已选工具列表：**
   - 显示用户已勾选的工具，按优先级排序
   - 每行显示：工具名称、分类标签、优先级序号、「配置」按钮、「移除」按钮
   - 用户可拖拽行调整顺序（优先级从高到低）

   - 用户点击已选工具行的「配置」按钮 → 展开参数配置面板

| 配置项 | 类型 | 说明 |

| 重试次数 | 数字输入 | 默认 0，失败后重试次数 |
| 参数映射 | 映射配置 | 将数字员工上下文变量映射到工具参数 |

5. 用户点击「保存」保存工具参数配置
6. **预览：** 用户可点击工具行的「预览」按钮 → 弹出测试对话框 → 输入参数 → 点击「测试调用」→ 显示工具返回结果

**反馈：** 勾选/取消勾选时右侧列表实时更新；测试调用结果在弹窗中展示

#### 3.2.2 Action 权限配置

**入口：** 创建/编辑向导 → 第二步「能力配置」→ 区域 B「Action 权限配置」

**操作步骤：**

1. 用户展开「Action 权限配置」折叠面板 → 显示 Action 选择区域
2. **Action 列表：**
   - 顶部搜索框 + 分类筛选（全部 / 数据操作 / 流程触发 / 通知 / 文件 / 系统管理）
   - 数据来源：从 TECH-ACTION 注册的 Action 中获取
   - 每个 Action 显示：名称、图标、分类、风险等级标签（低/中/高）、简短描述

3. **权限设置：**
   - 用户勾选 Action → 该行展开权限配置区域：

| 配置项 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| 权限级别 | 单选 | 仅查询 | 仅查询（只读）/ 查询+执行（可读可写）/ 完全控制（含删除） |
| 执行确认策略 | 单选 | 自动执行 | 自动执行 / 需人工确认 / 高风险需审批 |
| 数据范围限制 | 条件构建器 | 无限制 | 限制 Action 可操作的数据范围（如仅可操作本部门数据） |
| 调用频率限制 | 数字输入(次/分钟) | 无限制 | 限制 Action 调用频率 |

4. **风险等级说明：**
   - 低风险：查询类操作，可自动执行
   - 中风险：写入/修改类操作，默认需人工确认
   - 高风险：删除/审批/外部通知类操作，默认需审批

5. 用户可批量设置：勾选多个 Action → 点击「批量设置权限」→ 统一设置权限级别和执行策略

6. **已选 Action 汇总：** 底部显示已选 Action 表格

| 列 | 说明 |
|---|---|
| Action 名称 | 如"创建报销单""查询员工信息" |
| 分类 | 数据操作/流程触发/通知等 |
| 风险等级 | 低(绿色)/中(黄色)/高(红色) |
| 权限级别 | 仅查询/查询+执行/完全控制 |
| 执行策略 | 自动/需确认/需审批 |
| 操作 | 编辑 / 移除 |

**反馈：** 勾选 Action 时自动展开权限配置；权限级别变更时风险等级标签颜色同步更新

#### 3.2.3 RAG 知识库绑定

**入口：** 创建/编辑向导 → 第二步「能力配置」→ 区域 C「RAG 知识库绑定」，以及第三步「知识范围」

**操作步骤：**

1. 用户展开「RAG 知识库绑定」折叠面板 → 显示知识库选择区域
2. **知识库列表：**
   - 顶部搜索框 + 「刷新」按钮
   - 数据来源：从 TECH-RAG 已创建的知识库列表获取
   - 每行显示：知识库名称、文档数量、向量维度、最后更新时间、状态标签
   - 用户勾选知识库前的复选框进行绑定

3. **检索策略配置（绑定后展开）：**

| 配置项 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| 检索方式 | 单选 | 混合检索 | 纯向量检索 / 纯关键词检索 / 混合检索（向量+关键词） |
| Top-K | 数字输入 | 5 | 检索返回的文档片段数量，1-20 |
| 相似度阈值 | 滑块 | 0.7 | 0.0-1.0，低于阈值的文档不返回 |
| 重排序 | 开关 | 开启 | 开启后使用 reranker 模型对 Top-K 结果重排序 |
| 重排序模型 | 下拉选择 | 默认 reranker | 选择重排序模型（仅"重排序"开启时显示） |
| 上下文窗口 | 数字输入 | 4096 | 检索结果注入 LLM 上下文的最大 token 数 |

4. **知识过滤（可选）：**
   - 用户点击「设置知识过滤」→ 弹出条件构建器
   - 可设置仅检索特定标签、特定分类、特定时间范围的文档
   - 条件支持：等于/不等于/包含/范围

5. **绑定预览：**
   - 底部显示已绑定知识库的汇总信息：总文档数、总向量数、预计检索延迟
   - 用户可点击「测试检索」→ 输入查询语句 → 展示检索结果和相似度分数

**反馈：** 绑定/解绑知识库时汇总信息实时更新；测试检索结果在弹窗中展示

| Function Calling | 开关 | 开启 | 开启后支持工具调用 |

### 3.3 知识提炼

#### 3.3.1 上传文档

**入口：** 员工详情页 → 「知识库」Tab → 「上传文档提炼」按钮，或创建向导第三步的「上传文档提炼」按钮

**页面路径：** `/dw/employees/{employeeId}/knowledge/upload`

**页面布局：**

- **顶部说明栏：** 蓝色信息条"上传企业制度、流程文档或访谈录音，AI 将自动抽取 Ontology 概念、实体、规则与 Action"
- **上传区域：** 大面积拖拽上传区，虚线边框，hover 变蓝
- **文档列表：** 上传区域下方，表格展示已上传文档

**操作步骤：**

1. 用户进入知识提炼页面
2. 用户通过以下方式上传文档：
   - **拖拽上传：** 将文件拖入上传区域 → 系统自动上传
   - **点击上传：** 点击上传区域 → 弹出文件选择器 → 选择文件 → 点击「打开」
   - **批量上传：** 可同时选择多个文件

3. **上传时填写文档信息（每个文档）：**

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|---|---|---|---|---|
| 文档名称 | 文本输入 | 是 | 文件名 | 文档的显示名称 |
| 文档类型 | 下拉选择 | 是 | 自动识别 | 制度文档 / 流程文档 / 访谈记录 / 法规文件 / 其他 |
| 关联知识库 | 下拉选择 | 否 | 默认知识库 | 选择将知识写入的目标知识库 |
| 自动抽取 | 开关 | 是 | 开启 | 上传后是否自动触发 Ontology 抽取 |

4. **支持的文件格式：**

| 格式 | 说明 |
|---|---|
| PDF | 支持文字版和扫描版（扫描版自动 OCR） |
| Word (.docx/.doc) | 自动解析文本和表格 |
| TXT / Markdown | 直接解析文本 |
| 音频 (MP3/WAV/M4A) | 访谈录音，自动 ASR 转写后再抽取 |
| Excel (.xlsx) | 解析表格结构化数据 |

5. **文件限制：** 单文件最大 50MB，单次最多 20 个文件

6. 上传过程中显示进度条（每个文件独立进度），上传完成后：
   - 文件状态变为"已上传"
   - 如开启"自动抽取" → 状态变为"抽取中"（见 3.3.2）
   - 如关闭"自动抽取" → 状态为"待抽取"，用户可手动点击「开始抽取」按钮

7. **已上传文档列表：**

| 列 | 说明 |
|---|---|
| 文档名称 | 文件名 + 格式图标 |
| 文档类型 | 制度/流程/访谈/法规/其他 |
| 文件大小 | 如"2.3 MB" |

| 操作 | 开始抽取 / 查看抽取结果 / 重新抽取 / 删除 |

**反馈：** 上传成功 Toast 绿色提示"文档上传成功"；上传失败 Toast 红色提示"上传失败：{原因}"

#### 3.3.2 AI 自动抽取 Ontology

**入口：** 文档上传后自动触发（开启"自动抽取"时），或手动点击「开始抽取」按钮

**操作步骤：**

1. 系统收到抽取指令 → 调用 `POST /api/v1/dw/knowledge/documents/{documentId}/extract`
2. 抽取流程（后台执行，用户可见进度）：

   ```
   步骤 1/5: 文档解析
     → TECH-LLMGW 解析文档内容（PDF/Word/TXT → 纯文本）
     → 音频文档先 ASR 转写
     → 进度：0-20%

   步骤 2/5: 概念识别
     → AI 识别文档中的业务概念和术语
     → 如"报销单""审批人""金额上限""费用类别"
     → 进度：20-40%

   步骤 3/5: 实体与属性抽取
     → 识别实体类型及其属性
     → 如实体"报销单"具有属性：单号、申请人、金额、日期、类别
     → 识别实体间关系：报销单 → 属于 → 费用类别
     → 进度：40-60%

   步骤 4/5: 规则提取
     → 从制度文档中提取业务规则
     → 如"差旅费报销标准：高铁二等座，住宿 ≤500元/晚"
     → 如"审批规则：金额 >10000 需总经理审批"
     → 进度：60-80%

   步骤 5/5: Action 识别
     → 识别文档中涉及的业务操作
     → 如"提交报销""审批通过""退回报销"
     → 生成 Action 草稿（名称、参数、触发条件）
     → 进度：80-100%
   ```

3. 抽取过程中，文档列表中该文档的"抽取状态"显示为"抽取中"（蓝色旋转图标 + 当前步骤文字）
4. 抽取完成后：
   - 状态变为"已抽取"（绿色）
   - 操作列出现「查看抽取结果」按钮
   - Toast 提示"文档「{文档名称}」抽取完成，共识别 {N} 个概念、{M} 个实体、{K} 条规则、{L} 个 Action"
5. 抽取失败：
   - 状态变为"抽取失败"（红色）
   - 操作列出现「查看错误」和「重新抽取」按钮
   - Toast 红色提示"抽取失败：{错误原因}"

**抽取耗时预期：**
- 100 页以内文档：< 60 秒
- 音频文件（30 分钟以内）：< 90 秒（含 ASR）

| 元素 | 说明 |

| 来源段落 | 点击「定位」→ 右侧原文滚动到对应段落并高亮 |
| 置信度 | 高(绿色)/中(黄色)/低(红色) 标签 |
| 操作 | 编辑 / 删除 / 通过 |

   - 用户可逐个审核：点击「通过」标记为已确认，点击「编辑」修改概念名称和描述，点击「删除」移除
   - 用户可点击「添加概念」手动添加遗漏的概念

3. **实体审核：**
   - 展开「实体」折叠面板 → 显示抽取的实体类型
   - 每个实体类型显示：

| 元素 | 说明 |
|---|---|
| 实体名称 | 如"报销单" |
| 属性列表 | 如"单号、申请人、金额、日期、费用类别"，每个属性可编辑/删除 |
| 关系列表 | 如"报销单 → 属于 → 费用类别"，可编辑/删除 |
| 来源段落 | 点击「定位」→ 原文高亮 |
| 置信度 | 高/中/低 |
| 操作 | 编辑属性 / 添加属性 / 删除实体 / 通过 |

4. **规则审核：**
   - 展开「规则」折叠面板 → 显示提取的业务规则
   - 每条规则：

| 元素 | 说明 |
|---|---|
| 规则名称 | 如"差旅费报销标准" |
| 规则条件 | 如"费用类别 = 差旅 AND 交通工具 = 高铁" |
| 规则结果 | 如"报销上限 = 二等座票价" |
| 规则原文 | 提取自文档的原文段落 |
| 置信度 | 高/中/低 |
| 操作 | 编辑 / 删除 / 通过 |

   - 用户可点击「添加规则」手动添加遗漏的规则
   - 规则条件支持可视化条件构建器（字段 + 运算符 + 值）

5. **Action 审核：**
   - 展开「Action」折叠面板 → 显示识别的业务操作
   - 每个 Action：

| 元素 | 说明 |
|---|---|
| Action 名称 | 如"提交报销申请" |
| 描述 | 操作说明 |
| 输入参数 | 参数列表（名称、类型、是否必填） |
| 输出结果 | 预期返回结果 |
| 触发条件 | 如"用户请求报销时触发" |
| 置信度 | 高/中/低 |
| 操作 | 编辑参数 / 删除 / 通过 |

6. **批量操作：**
   - 「全部通过」按钮：将所有抽取项标记为已确认
   - 「全部拒绝」按钮：将所有抽取项标记为已拒绝
   - 用户也可使用多选勾选后批量通过/拒绝

7. **提交审核：**
   - 用户审核完成后，点击底部「提交审核」按钮
   - 系统校验：至少有 1 项已确认的内容
   - 校验通过 → 调用 `POST /api/v1/dw/knowledge/documents/{documentId}/extraction-result/approve`
   - 系统将审核通过的内容写入 TECH-ONT 本体引擎：
     - 概念 → 写入 Ontology Concept
     - 实体 → 写入 Ontology Entity Type + 属性
     - 关系 → 写入 Ontology Relation
     - 规则 → 写入 TECH-RULE 规则引擎
     - Action → 注册到 TECH-ACTION（草稿状态，需在 Action 管理中激活）
   - 写入成功 → Toast 绿色提示"审核通过，知识已写入 Ontology 本体引擎" → 状态变为"已入库"
   - 用户点击「保存草稿」→ 保存当前审核进度，不提交，状态保持"审核中"

**反馈：** 每个抽取项的通过/拒绝状态实时更新；定位原文时右侧自动滚动和高亮

#### 3.4.2 创建任务

| 字段 | 类型 | 必填 | 校验规则 | 默认值 | 说明 |
|---|---|---|---|---|---|
| 任务标题 | 文本输入 | 是 | 1-100 字符 | - | 如"Q3 财务数据分析报告" |

| 配置项 | 类型 | 默认值 | 说明 |
|---|---|---|---|

| 超时时间 | 数字输入(小时) | 24 | 任务超时时间，超时自动标记失败 |
| 完成通知 | 开关 | 开启 | 任务完成后通知分配者 |

6. 用户点击「创建并分配」按钮 → 系统校验：

7. 校验通过 → 调用 `POST /api/v1/dw/tasks` 创建任务
8. 创建成功 → Toast 绿色提示"任务已创建并分配给「{员工名称}」" → 任务列表自动刷新，新任务出现在列表中
9. 用户点击「取消」→ 关闭对话框

```
│   │ 耗时:5分│     │ 进度60% │     │         │              │
│   └─────────┘     └─────────┘     └─────────┘              │
│                                                             │
│   执行日志:                                                 │
│   14:30 数据分析师开始查询Q3数据...                          │
│   14:35 数据分析完成，结果已传递给报告撰写者                  │
│   14:35 报告撰写者开始撰写报告...                             │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│  [暂停协作] [查看详情] [取消协作]                             │
└─────────────────────────────────────────────────────────────┘
```

5. **实时可视化：**
   - 画布上的节点状态实时更新（颜色变化）
   - 已完成节点：绿色 + 耗时
   - 执行中节点：蓝色脉冲 + 进度
   - 待执行节点：灰色
   - 失败节点：红色
   - 节点间的连线动态高亮（数据流方向）

6. **执行日志：** 底部实时显示协作执行日志，包括：
   - 任务开始/完成事件
   - 任务委派事件（A2A 通信记录）
   - 结果传递事件
   - 异常和冲突事件

7. **用户操作：**
   - 「暂停协作」→ 暂停整个协作流程，当前执行中的任务完成后暂停
   - 「取消协作」→ 取消协作，弹出确认框
   - 点击节点 → 查看该节点的任务详情和执行轨迹

8. 协作完成 → 所有节点变为绿色 → 状态变为"已完成" → 弹出完成通知 → 可查看结果汇总（3.7.4）

#### 3.7.4 结果汇总

## 5. 依赖关系

### 5.1 上游依赖

| 上游服务 | 依赖内容 | 依赖类型 | 说明 |
|---|---|---|---|
| TECH-AGENT | Agent 框架、任务执行引擎、多步推理、ReAct 模式 | 强依赖 | 数字员工的核心执行引擎，所有任务通过 TECH-AGENT 编排执行 |
| TECH-RAG | 知识库检索、文档嵌入、混合检索、重排序 | 强依赖 | 数字员工的知识检索能力，RAG 知识库绑定后通过 TECH-RAG 检索 |
| TECH-ONT | Ontology 抽取写入、概念查询、知识图谱、实体管理 | 强依赖 | 知识提炼的写入目标，数字员工通过 Ontology 理解数据语义 |
| TECH-LLMGW | LLM 推理、对话生成、文档理解、ASR 转写 | 强依赖 | 所有 AI 推理通过 LLM Gateway，不直接调模型 API |
| TECH-A2A | A2A 协议、Agent 发现、跨 Agent 通信、任务委派 | 强依赖 | 多员工协作的通信基础，外部 Agent 协作的协议支持 |
| TECH-ACTION | Action 注册、执行、权限校验 | 强依赖 | 数字员工的 Action 权限配置和执行 |
| TECH-MCP | MCP 协议 Tools 注册和调用 | 中依赖 | 数字员工的 Tool 工具选择来自 MCP 注册的 Tools |
| TECH-IAM | 身份认证、权限控制、组织架构 | 强依赖 | 数字员工操作权限、用户身份验证 |
| TECH-OBS | 操作审计、分布式追踪、日志收集 | 中依赖 | 执行轨迹的 Trace 链路、操作日志记录 |
| TECH-RULE | 规则引擎、业务规则执行 | 中依赖 | 知识提炼中提取的规则写入 TECH-RULE |

### 5.2 下游消费方

| 下游方 | 消费内容 | 说明 |
|---|---|---|
| APP-DASHBOARD | 数字员工运行状态指标、任务统计、质量评分 | 仪表盘展示数字员工概览数据 |
| APP-SUPERAI | 数字员工对话能力复用 | SuperAI 可调用数字员工进行专业领域对话 |

| APP-APPHUB | 数字员工作为 Action 执行者 | 低代码应用中的流程节点可触发数字员工执行 |

### 5.3 交互流程

```
管理者创建数字员工
  → APP-DW 保存配置
  → 调用 TECH-AGENT 创建 Agent 实例
  → 调用 TECH-RAG 绑定知识库
  → 调用 TECH-LLMGW 注册模型配置
  ↓
知识工程师上传文档
  → APP-DW 调用 TECH-LLMGW 解析文档
  → AI 抽取 Ontology 草稿
  → 人工审核确认
  → 调用 TECH-ONT 写入本体引擎
  → Neo4j 图库存储 + Milvus 向量库索引
  ↓
用户/系统分配任务
  → APP-DW 创建任务记录
  → 调用 TECH-AGENT 启动任务执行
  ↓
数字员工执行任务
  → TECH-AGENT 编排多步推理（ReAct 模式）
    ├─ TECH-RAG: 向量检索 + 关键词检索 → 重排序 → 注入上下文
    ├─ TECH-LLMGW: 模型推理 → 生成回答/决策/计划
    ├─ TECH-ACTION: 执行业务操作（含权限校验）
    └─ TECH-MCP: 调用外部 MCP Tools（如需要）
  → 每步执行记录写入执行轨迹
  ↓
多员工协作
  → APP-DW 编排协作流程
  → TECH-A2A 实现 Agent 间通信
  → Agent Card 交换 → 任务委派
  → 结果汇聚汇总
  ↓
执行结果与评估
  → 结果持久化到 PostgreSQL
  → 对话记录自动评估
  → TECH-OBS 记录审计日志和 Trace
  → 生成效果报表和优化建议
```

### 5.4 数据流转

```
文档上传
  → TECH-LLMGW 解析（PDF/Word → 文本，音频 → ASR 转写）
  → 知识抽取（概念/实体/规则/Action）
  → 人工审核
  → TECH-ONT 写入本体 → Neo4j 图库（关系）+ Milvus 向量库（嵌入）
  ↓
任务执行
  → TECH-AGENT 编排
    ├─ TECH-RAG: 混合检索 → Reranker 重排序 → 上下文注入
    ├─ TECH-LLMGW: Prompt → 模型推理 → 输出
    ├─ TECH-ACTION: Action 执行 → 业务操作
    └─ TECH-MCP: MCP Tool 调用 → 外部能力
  → 执行轨迹 → PostgreSQL 持久化
  ↓
A2A 协作
  → TECH-A2A → Agent Card 交换 → 任务委派
  → 跨 Agent 通信 → 结果回传
  → 结果汇聚
  ↓
效果评估
  → 对话记录 → 自动评分（TECH-LLMGW 评估模型）
  → 人工抽检 → 标注数据
  → 评估统计 → 报表生成
  → 优化建议 → 反馈到配置优化
```

## 8. 非功能需求

### 8.1 性能需求

| 指标 | 要求 |
|---|---|
| 对话首字响应 | < 2 秒 |
| 任务分配延迟 | < 1 秒 |
| 文档抽取（100页以内） | < 60 秒 |
| 音频转写（30分钟以内） | < 90 秒 |
| 状态更新推送 | < 3 秒 |
| 并发数字员工 | 支持 100 个同时在线 |
| 协作任务委派延迟 | < 2 秒 |
| 执行轨迹查询 | < 1 秒 |
| 效果报表生成 | < 5 秒 |

### 8.2 可用性需求

| 指标 | 要求 |
|---|---|
| 可用性 | 99.9% |
| 数字员工故障恢复 | 自动重试 + 故障切换 |
| 任务持久化 | 任务状态持久化到 PostgreSQL，支持断点恢复 |
| A2A 通信容错 | 网络中断后自动重连，3 次重试 |
| WebSocket 断线重连 | 自动重连，指数退避 |

### 8.3 安全需求

- 数字员工操作权限通过 TECH-IAM 严格控制
- 所有 Action 执行需权限校验
- 对话内容敏感信息脱敏（身份证、银行卡、手机号等）
- 文档上传需病毒扫描和大小限制
- A2A 外部 Agent 通信需身份验证和加密（mTLS）
- 操作审计日志通过 TECH-OBS 记录，保留 90 天
- 高风险操作需人工确认或审批

### 8.4 AI 质量需求

| 指标 | 要求 |
|---|---|
| Ontology 抽取准确率 | > 75%（人工审核后 100%） |
| 任务执行成功率 | > 90% |
| 对话质量自动评分准确率 | > 80% |
| A2A 协作成功率 | > 95% |
| 知识检索相关性 | Top-5 准确率 > 85% |

### 8.5 可扩展性

- 支持自定义数字员工模板
- 知识抽取支持插件化文档解析器
- 协作流程支持可视化编排
- 支持 A2A 协议对接外部 Agent
- Tool 和 Action 支持动态注册
- 检索策略支持自定义扩展

---

> 本 PRD 遵循用户动线设计，每个功能点到按钮级操作。开发时按 Phase 顺序增量交付，每个 Phase 可独立验证。所有 LLM 调用通过 TECH-LLMGW，RAG 检索通过 TECH-RAG，Agent 执行通过 TECH-AGENT，跨 Agent 协作通过 TECH-A2A，不直接调用底层模型 API 或基础设施。
