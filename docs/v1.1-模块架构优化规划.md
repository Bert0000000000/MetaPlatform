# MetaPlatform v1.1 模块架构优化规划

> 基于百特搭产品设计理念 + 平台当前实现，重新规划前端模块架构

---

## 一、百特搭核心设计理念提炼

| 维度 | 百特搭做法 | MetaPlatform 借鉴点 |
|------|-----------|-------------------|
| **产品定位** | NoCode + LowCode + ProCode 三位一体 | 业务人员零代码、建模师低代码、开发者全代码 |
| **核心驱动** | 元数据模型驱动一切 | 本体 = 单一真源，驱动 UI/流程/数据/权限 |
| **模块划分** | 表单 → 流程 → 数据 → 集成 → 开发 | 按用户操作场景划分，非技术架构划分 |
| **AI 融合** | AI 能力贯穿全平台 | AI 是基质，不是独立模块 |
| **沙盒环境** | 设计态/运行态分离 | 建模 → 预览 → 发布 → 运行 四态分离 |
| **SaaS 能力** | 多租户 + 多版本 + 灰度发布 | 租户隔离 + 应用版本管理 |

---

## 二、当前 1.0 问题诊断

### 问题 1：模块按技术层划分，非用户场景
```
1.0 侧边栏：建模特工场 / 页面设计器 / 流程设计器 / AI对话 / 能力中心 / 集成中心 / ...
→ 用户视角：我要做一个"请假审批"，需要跨 5 个页面
→ 百特搭做法：在"应用"中一站式完成 表单+流程+数据+权限
```

### 问题 2：缺少"应用"概念
```
1.0：ObjectType 是散落的，没有"应用"包裹
百特搭：一切都在"应用"中，应用 = 表单 + 流程 + 数据 + 权限 + 发布
```

### 问题 3：AI 不够突出
```
1.0：AI 对话是独立页面，与建模/流程/数据割裂
百特搭：AI 能力嵌入每个操作环节
```

### 问题 4：缺少 Dashboard 入口
```
1.0：首页是空的页面列表
百特搭：首页是应用总览 + 待办 + 数据看板
```

#### 模块 1：🏠 工作台（Dashboard）

**定位**：用户进入平台的第一屏，提供全局视图和快捷操作

**页面结构**：
```
工作台
├── 顶部统计卡片
│   ├── 应用总数 / 进行中流程 / 今日活跃 / 待办数
│   └── 点击跳转到对应模块
├── 待办事项
│   ├── 我的审批（来自流程中心）
│   ├── 我的工单（来自应用中心）
│   └── AI 建议（来自 SuperAI）
├── 最近访问
│   ├── 最近打开的应用
│   ├── 最近编辑的表单
│   └── 最近查看的数据
├── 快捷入口
│   ├── 新建应用
│   ├── 新建表单
│   ├── AI 建模
│   └── 数据导入
└── 数据看板（可配置）
    ├── 业务指标卡片
    ├── 趋势图表
    └── 告警提醒
```

**对应后端**：
- ObjectType 数量统计 → `/api/v1/object-types?tenantId=...`
- 流程待办 → `/api/v1/processes/instances?assignee=current&status=running`
- Agent 执行 → `/api/v1/agents/executions?userId=current`
- 审计日志 → `/api/v1/security/audit-logs?limit=10`

## 八、与百特搭的关键差异

| 维度 | 百特搭 | MetaPlatform |
|------|--------|-------------|
| 核心驱动 | 元数据模型 | **本体论（Ontology）** |
| AI 能力 | AI 辅助 | **AI 基质（贯穿全栈）** |
| 数据能力 | BI 报表 | **数据湖 + 数仓 + MDM + 血缘** |
| 知识能力 | 无 | **RAG + GraphRAG + 知识库** |
| 流程能力 | 流程中台 | **BPMN + AI 决策节点 + 流程挖掘** |
| 部署能力 | 公有云/私有云 | **K8s + Docker + 信创** |

MetaPlatform 的核心差异化：**本体论驱动 + AI 基质 + 数据栈内置**，这是百特搭不具备的。
