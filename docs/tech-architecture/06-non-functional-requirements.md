# §6 非功能需求（NFR）

> **目标**：把 spec v1.2 中"软目标"（"性能好"、"可扩展"）转化为可测量的 NFR 指标。
> **对应 spec**：v1.2 §12 风险与挑战 + 路线图
> **状态**：v1.0 草稿

---

## §6.1 NFR 分类

```
NFR
├── 性能 (Performance)
│   ├── 响应时间 (Latency)
│   ├── 吞吐量 (Throughput)
│   └── 资源利用率 (Resource Utilization)
├── 可用性 (Availability)
│   ├── 服务可用性 (Uptime)
│   └── 恢复能力 (Recovery)
├── 扩展性 (Scalability)
│   ├── 水平扩展 (Horizontal)
│   └── 垂直扩展 (Vertical)
├── 安全性 (Security)
│   ├── 认证 (Authentication)
│   ├── 鉴权 (Authorization)
│   └── 加密 (Encryption)
├── 可维护性 (Maintainability)
│   ├── 可观测 (Observability)
│   ├── 可测试 (Testability)
│   └── 可部署 (Deployability)
├── 合规性 (Compliance)
│   ├── 国内合规
│   ├── 信创合规
│   └── 行业合规
└── 兼容性 (Compatibility)
    ├── 浏览器兼容
    ├── 信创兼容
    └── 第三方集成
```

|---|---|---|---|

| **LLM 服务故障** | N/A | < 5min | 多模型路由 |

### 6.3.3 优雅降级

| 故障 | 降级策略 | 用户感知 |
|---|---|---|
| **LLM 不可用** | 用规则引擎 | NL 问答失败，规则可用 |
| **RAG 不可用** | 用 ES 全文 | 召回质量下降 |
| **Doris 不可用** | 用 PG 临时查询 | 性能差但能用 |
| **Neo4j 不可用** | 用 PG 关系表 | 性能差但能用 |
| **Milvus 不可用** | 用 ES 全文 | 召回质量下降 |
| **Kafka 不可用** | Outbox 暂存 | 写入延迟增加 |

## §6.10 v0.1 plan 与本节的关系

| NFR 类别 | v0.1 覆盖 | 备注 |
|---|---|---|
| **性能** | ❌ 不覆盖 | v0.1 只追求跑通 |
| **可用性** | ❌ 不覆盖 | v0.1 docker-compose 单机 |
| **扩展性** | ❌ 不覆盖 | v0.1 单租户 |
| **安全性** | ❌ 不覆盖 | v0.1 硬编码 X-Actor-Id |
| **可维护性** | ⚠️ 部分（stdout 日志）| v0.2 完善 |
| **合规性** | ❌ 不覆盖 | v0.1 内部 Spike |
| **兼容性** | ❌ 不覆盖 | v0.1 单一环境 |

**结论**：v0.1 plan 不需要重写（NFR 是 v0.2+ 关注点）。

**v0.2 plan 必须补的 NFR**（按优先级）：
1. **认证 + 鉴权**（商用前提）
2. **多租户**（SaaS 前提）
3. **可观测**（运维前提）
4. **Outbox 模式**（数据一致性）
5. **限流 / 熔断**（稳定性）
6. **备份 / 恢复**（灾备）
