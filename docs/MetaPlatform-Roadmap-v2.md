> **文档版本**：v2
> **修订日期**：2026-07-10
> **取代**：v1.0 的「Phase 1 P0 / Phase 2 P1 / Phase 3 P2」时间表
> **核心原则**：**功能闭合 + 里程碑节奏 + 每个版本对齐百特搭某个能力板块**
> **配套依据**：
> - 《MetaPlatform 菜单产品设计 v3.1》

## 〇、为什么重做 Roadmap

### 0.1 v1.0 Roadmap（已作废）存在的 4 个问题

| # | 问题 | 具体表现 |
|---|------|----------|
| 1 | **节奏与容量错配** | Phase 1 把 209 周塞进 6-9 个月，根本跑不完 |
| 2 | **优先级反客为主** | AI 差异化、数字员工、AI 基质都被压在 P1/P2 后段 |
| 3 | **没有"对外窗口"** | 只有 Phase 1 末尾才能对外 demo，中间无任何可发布物 |
| 4 | **商业化被排除在主线** | 计费、SaaS 分级、订阅都在 P1，导致前 6-9 个月始终是裸奔形态 |

### 0.2 v2 Roadmap 的 5 条新原则

| 原则 | 含义 |
|------|------|
| **1. 按能力闭合切片** | 每个版本交付一个独立的、面向客户的能力包（如"基础表单能力 v0.1"、"BPMN 2.0 完整能力 v0.4"），而不是按 P0/P1 平铺 |
| **2. 每个版本有对外 demo** | 每个版本都至少能完成一个完整的"用户场景旅程"，并能被看到、跑通 |
| **3. AI 差异化中期发力** | v2.0 之前专注对标百特搭，v2.0 起把 AI 基质 / 数字员工 / GraphRAG 推到前台 |
| **4. 计费 v1 才补** | v1 之前商业形态用"申请试用"，v1 起切换为正式 SaaS 分级计费 |
| **5. 时间不是主轴** | 不锁季度，每个版本"能交付"就发；粗估时间只是参考 |

### 0.3 版本号规则

格式：`v[major].[minor].[patch]`（每段都对应不同的工作流）

**重大决策（2026-07-10 确定）**：用三层版本号，而不是传统两层。三层分别对应：
- **major**：里程碑式变化，重大能力区或商业模式升级
- **minor**：对应"能力闭合"的某个完整切片（每个 minor 必须可独立 release）
- **patch**：仅做 bug 修复（hotfix）

实际使用：

| 路线 | major.minor | 对应内容 |
|------|-------------|----------|
| v1.0.0 (GA) | 已发 | 当前稳定版 |
| **v1.0.1 / v1.0.2 / ...** | 进行中 | **应用中心 MVP 主线，按能力切片逐个交付**（文档落地在 [`v1.0.x/`](v1.0.x/README.md)） |
| **v1.1.0** | 未来 | SuperAI + 数字员工 + Deerflow 集成 |
| **v1.2.0** | 未来 | 数据中心加强 + 本体引擎 + 知识图谱 |
| v1.x.y (patch) | 滚动 | hotfix，仅修 bug |

举例说明：

- **v1.0.1** = 「应用中心 MVP：应用 + 表单 + 简单流程」（首个交付中的 minor）
- **v1.0.2** = 「应用中心增加：关联对象 + 高级列表 + 导入导出」（下个 minor）
- **v1.1.0** = 「第二阶段发力：AI + 数字员工」（重大能力增加，升 major）
- **v1.1.1** = 「在 v1.1.0 上的 hotfix 修某个 bug」

注意：
- "v1.0.6 完成 = v1.0 GA 商业级" 的提法只用于示意，**实际交付通过逐个 v1.0.x minor 来渐近达到**
- 旧文档里"v1.0 = 商业级 GA"已被**新版"v1.0.x"代替**，这是范式转变

详细工作流见 [`v1.0.x/06-git-branching-and-release.md`](v1.0.x/v1.0.1/06-git-branching-and-release.md)。

## 七、不做的事情（Out of Scope，明确说"不做"）

## 十一、FAQ（决策时的常见疑问）

### Q1：为什么不锁季度，硬要按能力交付？

A：因为里程碑节奏能避免"为了凑时间上线半成品"。如果 v0.4 BPMN 2.0 卡住了，硬上季度版本反而伤害客户。能力闭合才发布，每个版本都对得起"对外 demo"的承诺。

### Q2：v0.1 真的只需要 6-8 周吗？

A：是的，因为 v0.1 是"我已经落地的本体引擎 + 页面生成器 + 前端"的渐进式组合。核心是 UX 串通，不是开发新功能。

### Q3：AI 能力推后到 v2.0，会不会丢掉差异化？

A：不会。

- v1.0 之前产品打磨稳定性和基础能力，反而是 AI 工程师最缺的"地面"
- v2.0 一次性把 AI 基质 + 数字员工 + AI 助手浮窗推出，对外感受会更强烈
- 提前半年做 AI 是"边造飞机边换引擎"，风险更高

### Q4：计费为什么不是 v0.1 就有？

A：v0.x 阶段是白名单试用，没必要做复杂计费。客户不需要为"还没做完的功能"付费。v1.0 才是"商业级 GA"——产品完成度足够支撑付费转化。

### Q5：CODE EXPORT 等到 v1.2，会不会错过客户窗口？

A：不会。百特搭的 CODE EXPORT 是"锦上添花"，不是"刚需"。v1.0 先把"在线可用的应用 + 流程 + 数据"做稳，再给"导出代码做线下集成"的选项，正好对应中型 → 大型客户的拓展节奏。
