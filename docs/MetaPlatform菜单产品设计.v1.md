# MetaPlatform 菜单产品设计 v1

> **文档版本**：v1
> **作者**：MetaPlatform 产品团队
> **核心目标**：把"百特搭架构图"和"v1.2 模块架构规划"落到**可操作、可开发、可落地**的菜单结构
> **设计理念**：**双层菜单 × 4 类角色 × 业务场景驱动**
> **更新日期**：2026-07-03

---

## 〇、为什么 v1.2 的菜单设计不好？

v1.2 把 12 个模块平铺成侧边栏一级菜单，存在 4 个致命问题：

| 问题 | 后果 |
|------|------|
| ❌ **菜单项过多** | 12 个一级菜单，浏览器侧边栏需要滚动，用户认知负担过重 |
| ❌ **缺少业务场景视角** | 用户要"请假"得跨 5 个菜单（架构中心建对象→应用中心配置→流程中心画流程→数据中心配权限→平台配置发版），割裂感强 |
| ❌ **AI 能力散落** | 智能问数在"决策引擎"，NL 建模在"SuperAI"，AI 测试在"质量验证平台"，同一个 AI 能力被切成 3 块 |
| ❌ **角色差异没体现** | 领导只需要"看"，却要面对"开发运维"的所有菜单 |

**v1 设计哲学**：**用户不是按"模块"思考的，是按"我现在要干什么"思考的**。

## 一、v1 设计哲学：双层菜单 × 4 类角色

### 1.1 一句话总结

> **左：3 大场景空间（侧边栏，固定 3 项）**
> **右：场景内 Tab 切换（顶部，动态加载）**
> **全局：⌘K AI 助手（任何页面唤起）**

### 1.3 一级菜单（左侧侧边栏）：3 大场景空间（固定）

把"百特搭架构图"的所有 45 个子模块，**重新归类为 3 大场景**：

| 场景空间 | 一句话 | 适合角色 | 包含模块 |
|---------|--------|---------|----------|
| 🏠 **我的工作** | 我个人要做的事 | 所有角色 | 数字工作台、SuperAI、我的应用、我的数字员工、消息中心 |
| 🏗️ **构建应用** | 我要给团队/企业交付的能力 | 业务 + 开发 | 架构中心、融合构建引擎、应用中心、应用市场 |
| 🧠 **数据智能** | 我要让数据和 AI 产生价值 | 业务 + 开发 + 领导 | 决策引擎、多智能体协作、数据中心、质量验证平台 |

> **关键设计**：运维类功能（管理端、平台配置）**不进入一级菜单**，而是放进右上角"齿轮"或用户头像下拉，避免污染主场景流。

### 1.4 二级菜单（顶部 Tab）：场景内功能

进入"🏗️ 构建应用"后，顶部出现 Tab 切换：
```
[ 应用 ] [ 架构 ] [ 流程 ] [ 页面 ] [ 智能体 ] [ 模板市场 ]
```
进入"🧠 数据智能"后，顶部 Tab：
```
[ 问数 ] [ 指标 ] [ 决策 ] [ 数据源 ] [ 数据湖仓 ] [ 知识 ] [ 质量 ]
```

## 二、3 大场景空间详细设计

### 🏠 场景 1：我的工作（My Workspace）

**核心理念**：千人千面，每个角色登录后看到的"我的工作"完全不同。

```
我的工作
├── Tab 1: 工作台（按角色动态变化）
│   ├── 👑 领导版：AI 看板 + 战略指标 + 待审批 + 团队数字员工
│   ├── 💼 业务版：我的待办 + 最近应用 + 我的数字员工
│   ├── 🔧 开发版：开发中的应用 + 开发中的智能体 + FDE 入口
│   └── ⚙️ 运维版：系统健康 + 告警 + 资源使用率
│
├── Tab 2: 我的应用（消费侧）
│   ├── 已安装应用
│   ├── 最近使用
│   ├── 收藏应用
│   └── 团队应用推荐
│
├── Tab 3: 我的数字员工（消费侧）
│   ├── 我创建的数字员工
│   ├── 分配给我的数字员工
│   ├── 数字员工对话历史
│   └── 使用统计
│
├── Tab 4: 消息中心
│   ├── 待办事项（流程/审批/工单）
│   ├── @我的消息
│   ├── 系统通知
│   └── 数字员工主动推送
│
└── Tab 5: SuperAI（个人 AI 工作台）
    ├── 对话历史
    ├── 我的 AI 任务
    ├── Token 用量
    └── 模型偏好设置
```

**为什么这样设计**：
- 把"工作台、我的应用、消息中心"这些**消费侧**能力放在一起，符合用户直觉
- "数字员工"作为独立 Tab，强调 MetaPlatform 差异化
- "SuperAI"是个人 AI 助手，与各业务场景的 AI 嵌入是**两个层级**

- **应用 Tab 在最前**：用户进入"构建应用"第一眼看到"我要做什么应用"
- **架构 Tab 第二位**：建模是基础，但作为 Tab 不是主菜单
- **流程、页面、智能体、模板** 作为同层 Tab，体现"一个应用 = 对象+流程+页面+智能体"的设计
- **彻底告别"百特搭架构图"的 L2 引擎层划分** —— 把"融合构建引擎"的能力拆到"页面 Tab"内

### 🧠 场景 3：数据智能（Data & AI）

**核心理念**：把"数据中心"和"AI 决策"合并，让数据/AI 不再是"后台"，而是"驾驶舱"。

```
数据智能
│
├── Tab 1: 智能问数（驾驶舱）
│   ├── 自然语言提问（NL→SQL）
│   ├── 结果可视化（自动选图表）
│   ├── 追问与下钻
│   ├── 收藏与分享
│   └── 问数历史
│
├── Tab 2: 指标中心
│   ├── 我的指标
│   ├── 原子指标（基于业务对象）
│   ├── 派生指标（公式组合）
│   ├── 维度定义
│   ├── 指标血缘
│   └── 指标版本管理
│
├── Tab 3: 决策推送（业务推动）
│   ├── 异常告警
│   │   ├── 异常指标 → 自动建工单
│   │   └── 阈值配置
│   ├── 决策建议（AI 推荐）
│   ├── 复盘报告
│   └── 我的决策记录
│
├── Tab 4: 数据源
│   ├── 连接器列表（REST/DB/CSV/Kafka）
│   ├── 新建连接器
│   ├── 同步策略（全量/增量/定时）
│   ├── 字段映射
│   ├── 冲突解决
│   └── Webhook 端点
│
├── Tab 5: 数据湖仓
│   ├── 数据仓库（Doris）
│   │   ├── 表管理
│   │   ├── SQL 查询
│   │   ├── 仪表盘
│   │   └── 数据探索
│   ├── 数据湖（Hudi）
│   │   ├── Hudi 表管理
│   │   ├── 记录写入/读取
│   │   └── Schema 演进
│   └── 主数据 MDM
│       ├── 黄金记录
│       ├── 数据匹配
│       └── 合并/拆分
│
├── Tab 6: 知识库
│   ├── 文档管理
│   ├── 文档分块 + 向量化
│   ├── 知识搜索（RAG）

    ├── 质量评分（4 维）

    ├── 异常检测
    ├── 修复建议
    └── 数据血缘图谱
```

**关键设计点**：
- **问数 Tab 在最前**：体现"数据驾驶舱"理念，问数是一等公民
- **指标、决策、数据源、湖仓、知识、质量** 7 个 Tab，覆盖完整数据栈
- **融合百特搭"决策引擎"+"数据中心"+"知识库"** 为统一场景

## 三、4 类角色看到的菜单差异

### 角色定义

| 角色 | 典型身份 | 关注点 |
|------|---------|--------|
| 👑 领导（Executive） | CEO/CTO/部门负责人 | 看数据、做决策 |
| 💼 业务（Business User） | 业务人员/PM | 用应用、跑流程、看报表 |
| 🔧 开发（Developer） | 建模师/全栈/低代码开发 | 建模、开发、构建 |
| ⚙️ 运维（Ops） | 系统管理员/SRE | 部署、监控、权限 |

### 各角色看到的菜单

#### 👑 领导版

```
🏠 我的工作       ✅ 完整（领导版工作台）
🏗️ 构建应用       ⚠️ 只读（看应用列表、流程列表）
🧠 数据智能       ✅ 完整（重点使用）
                  └─ 问数 + 指标 + 决策 + 知识 + 质量
齿轮菜单：个人设置
```

#### 💼 业务版（默认登录角色）

```
🏠 我的工作       ✅ 完整（业务版工作台 + 消息中心）
🏗️ 构建应用       ⚠️ 部分（看应用、用应用、低代码）
                  └─ 应用 Tab（用） + 流程 Tab（跑流程） + 模板市场（安装模板）
🧠 数据智能       ✅ 部分（看数据）
                  └─ 问数 + 指标 + 决策 + 知识
齿轮菜单：个人设置
```

#### 🔧 开发版

```
🏠 我的工作       ✅ 完整（开发版工作台）
🏗️ 构建应用       ✅ 全权限
                  └─ 所有 6 个 Tab：应用 / 架构 / 流程 / 页面 / 智能体 / 模板市场
🧠 数据智能       ✅ 全权限
                  └─ 所有 7 个 Tab：问数 / 指标 / 决策 / 数据源 / 湖仓 / 知识 / 质量
齿轮菜单：个人设置
```

#### ⚙️ 运维版

```
🏠 我的工作       ✅ 完整（运维版工作台）
🧠 数据智能       ⚠️ 只读（看监控）
齿轮菜单（含全部管理功能）：
  ├── 平台配置（→ 旧 v1.1 平台配置）
  ├── 管理控制台（→ 新管理端）
  │   ├── 组织架构 + 权限
  │   ├── 插件管理
  │   ├── 服务网关
  │   └── 监控及日志
  ├── 部署及环境（单机/集群/私有化/沙盒）
  ├── 数据备份与恢复
  └── 系统设置
```

## 七、设计原则总结

### 7.1 7 条菜单设计原则

| 原则 | 说明 |
|------|------|
| **1. 场景驱动，非模块驱动** | 用户按"我要干什么"思考，不按"系统怎么分模块"思考 |
| **2. 一级菜单 ≤ 5 个** | 3 大场景 + AI 命令 + 用户菜单，认知负担最小 |
| **3. 二级菜单 ≤ 8 个** | Tab 形式展示，超过 8 个考虑再分组 |
| **4. 角色化分发** | 不同角色看到不同菜单，不是同一菜单不同权限 |
| **5. AI 内嵌而非独立菜单** | AI 助手 ⌘K 全局可唤起，同时每个页面右下角浮窗 |
| **6. 管理功能进齿轮** | 平台配置/管理控制台等"系统级"功能不污染主菜单 |
| **7. 数字员工是一等公民** | 单独 Tab/菜单，强调差异化 |

### 7.2 与 v1.2 菜单对比

| 维度 | v1.2 | v1（本设计） |
|------|------|------------|
| 一级菜单数 | 12 个（侧边栏） | **3 个 + 齿轮** |
| 二级展现形式 | 多级嵌套 | **顶部 Tab 切换** |
| 角色差异 | 同一菜单不同权限 | **不同角色不同菜单** |
| AI 入口 | 独立 SuperAI 模块 | **⌘K 全局 + 页面浮窗** |
| 管理功能污染 | 占一级菜单 | **进齿轮菜单** |
| 用户认知路径 | "请假" → 跨 5 菜单 | "请假" → 我的工作 → 我的应用 → 打开 |

## 八、验收标准

- [ ] 任何角色登录后，**主菜单不超过 3 个 Tab + 齿轮**
- [ ] ⌘K 唤起 AI 命令面板，200ms 内显示
- [ ] 领导角色看不到"页面生成器"等开发菜单
- [ ] 业务用户能在一个场景内完成"请假申请"全流程（不用切到构建应用）
- [ ] 开发人员能在一个场景内完成"建模 → 流程 → 页面 → 智能体 → 发布"全流程
- [ ] 数字员工有独立入口（我的工作 Tab + 构建应用 Tab 都有）
- [ ] 平台配置不进一级菜单
