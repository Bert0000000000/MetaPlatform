## 📋 版本号规则（v1.0.x 系列内部）

| 版本号 | 含义 | 频率 | Commit 策略 |
|--------|------|------|------------|
| **v1.0.1** | 第一个 minor 版本（应用 + 表单 + 基本流程） | 一次性发布 | 在 `release/v1.0.1` 分支完成后打 tag `v1.0.1` |
| **v1.0.2** | 第二个 minor 版本（应用 + 列表 + 模型） | 一次性发布 | 同上 |
| **v1.0.x** | 第 x 个 minor 版本 | 每个小版本一个 release 分支 | 每个 minor = 一个 tag |
| **v1.0.x.y** | 第 y 个 patch 版本（仅 bug fix） | 临时 | hotfix 后合并回主线 |
| **v1.1.0** | 第二个 major 版本启动 | 另起分支 | 主线进入 v1.1.x |

## 🔗 Git 工作流

每个 minor 版本（v1.0.1, v1.0.2, ...）遵循以下流程：

```
main (stable) ─────────────┬── main 保持随时可发布
                          │
release/v1.0.1            ┴── 开发分支，本 minor 完成后合并回 main
├── feature/app-shell      ┌── subagent 或开发者独立特性分支
├── feature/form-mvp
├── feature/process-mvp
└── docs/v1.0.1-xxx
```

### 提交规范

- **feature/**: 子特性分支，最终以 PR 合入 release/v1.0.x
- **release/v1.0.x**: 一个 minor 版本一条分支，封板后合并 main 并打 tag
- **main**: 仅是 release 分支的合并目标，CI 通过后才允许直接 push
- **docs/**: 文档与代码同分支

|--------|-------------------|------|
