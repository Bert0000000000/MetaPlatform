# v1.0.x 全系列概览

>
> **子文档**：本文档说明全系列总览；每个 minor 版本在子文件夹内有独立 README。

| 编号 | 决策 | 原因 |
|------|------|------|
| ADR-001 | v1.0.1 复用 `metaplatform-ontology-engine` + `metaplatform-page-generator` | 避免重复造轮子，已落地可直接调用 |
| ADR-002 | v1.0.1 新建 `metaplatform-app-service` | 应用中心要做领域编排，必须独立服务，否则后期加 v1.1 AI 时主 API 会膨胀 |

## 八、FAQ

### Q1：为什么不让 v1.1.0 提前启动？

A：v1.1.0 的核心是 AI 推理底座（LLM Gateway、Agent Runtime、Deerflow 集成），依赖 v1.0.6 的"应用中心数据模型"完整稳定。如果提前启动，AI 会基于不稳定的数据契约工作，导致 v1.1.0 返工。

### Q2：v1.1.0 必须用 Deerflow 吗？

A：是用户明确要求。Deerflow（深度研究型 Agent 框架）适合"AI 建应用"这种长流程、多步骤、需要工具调用的场景。v1.1.0 会用 Deerflow 作为 Agent Runtime 的核心框架。

### Q3：v1.2.0 和 v1.1.0 哪个先做？

A：**v1.1.0 先做**。理由：
- AI 是差异化卖点，先做出对客户的吸引力
- 数据中心加强 + 本体引擎 + 知识库可以放 v1.2.0 与 v1.1.0 并行启动（在 2027-Q2 末并行）
- 实际上 v1.1.0 的某些能力（如 NL2SQL）也会用 v1.2.0 的"加强数据中心"，所以 **v1.2.0 的部分能力可以在 v1.1.0 期间并行做**

### Q4：subagent 模型怎么协同？

A：详见 v1.0.1 sprint plan 第六章。原则是：
- **主线程**：做架构设计 + 关键决策 + 文档
- **subagent**：做独立可验证的特性模块（如 form-builder 组件、list-renderer 组件）
- 每个 subagent 完成后由主线程集成测试
