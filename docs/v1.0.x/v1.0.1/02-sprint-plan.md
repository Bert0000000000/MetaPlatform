## 七、subagent 协同开发方案

> 用户明确要求"可以采用 subagent 模型协同开发，提高效率"。下述方案是我们怎么落地。

### 7.1 总原则

| 项 | 主线程负责 | subagent 负责 |
|----|-----------|----------------|
| 架构决策 | ✅ | ❌ |
| 跨 sprint 任务 | ✅ | ❌ |
| 跨服务集成 | ✅ | ❌ |
| 独立特性模块 | ❌ | ✅（在主线程拆好的接口规范内） |
| 单测 | ❌ | ✅ |
| 自验证 | 主线程整合 | ✅ 完成自验证 |

### 7.2 哪些任务适合用 subagent

✅ **适合**（建议用 subagent）：
- 单文件功能（如某个 React 组件、单个 API 路由）
- 已有明确接口契约的后端实现（如 T1-1、T2-3）
- 标准库拼装的页面（如列表页 TS 实现）
- 单测大量生成（覆盖率冲刺）

❌ **不适合**（必须主线程）：
- 关键决策（架构、接口契约、性能优化）
- 跨服务集成（ontology-engine / page-generator 适配）
- 跨 sprint 联调（容易漏上下文）
- 安全相关（字段权限、跨用户隔离）

### 7.3 subagent 调度模板（每次都用同款）

```markdown
# Task: 实现 [任务 ID] [任务名]
## 上下文
- 版本：v1.0.1
- Sprint：[sprintN]
- 接口契约：[docs/v1.0.x/v1.0.1/contract/openapi.yaml 的具体路径]
- 涉及文件：[列出具体文件路径]
- 已落地接口：[相关 API 状态]

## 输入
- 上游 API: [接口路径 + 方法]
- 下游影响: [需要修改/扩展的其他文件]

## 输出要求
- 代码改动：[列出关键函数/文件]
- 验证方式：[具体怎么测试]
- 自测试：[至少 3 个用例的脚本]

## 不要做的事
- 不要修改接口契约
- 不要动 ontology-engine / page-generator
- 不要碰 sprint 范围之外的任务

## 完成后请告知
- 改了哪些文件（diff）
- 跑了什么测试（结果）
- 是否需要主线程对接 ontology-engine / page-generator
```
