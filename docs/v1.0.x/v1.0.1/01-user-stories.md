## 一、Epic 拆分（按用户场景）

v1.0.1 拆为 **3 个 Epic（里程碑）**，每 Epic 完成有明确的 demo：

| Epic | 标题 | 主要用户故事 |
|------|------|-------------|
| **M1** | 「应用骨架」| 业务能创建应用、配置对象、生成数据库表 |
| **M2** | 「表单 + 列表」| 业务能画表单、能填表单、能查列表 |
| **M3** | 「简单流程」| 业务能画 2 节点审批、能发起、能审批 |

每个 Epic 完成后 = 一次 demo 节点 + 一次发版内发布（如 alpha → beta → rc）。

## 二、Epic M1：应用骨架（用户故事与 AC）

### US-101：登录并进入应用中心

**作为** 一个已注册业务用户
**我希望** 登录后能进入"应用中心"看到我的应用列表
**以便于** 我能管理所有应用

#### 验收标准（Given/When/Then）

```gherkin
Given 我访问 http://localhost:5173/apps
When 我输入用户名 demo / 密码 demo1234 并提交
Then 我被重定向到 /apps
And 我能看到应用列表页（包含 demo 应用"订单管理"作为种子数据）
And 页面上有"新建应用"按钮
```

#### 详细 AC

| # | 条件 |
|---|------|
| AC-101.1 | 登录接口调用成功，token 存于 localStorage |
| AC-101.2 | 未登录访问 /apps 自动跳转 /login |
| AC-101.3 | 登录后显示用户名和头像 |
| AC-101.4 | 应用列表为空时显示空状态插画 + "新建应用" CTA |

---

### US-102：新建应用

**作为** 业务用户
**我希望** 能创建一个名为"出差报销"的应用
**以便于** 我在该应用中配置我的报销对象

#### 验收标准

```gherkin
Given 我登录且在 /apps 页面
When 我点击"新建应用"按钮 → 弹出对话框 → 输入名称"出差报销" → 选择 icon → 提交
Then 弹窗关闭，新应用出现在列表顶部
And 我能在数据库 app 表看到一条新记录
```

#### 详细 AC

| # | 条件 |
|---|------|
| AC-102.1 | 应用名必填，长度 2-30 |
| AC-102.2 | 应用 icon 从预设库选择（至少 12 个） |
| AC-102.3 | 创建成功后跳转到该应用的"对象管理"页 |
| AC-102.4 | 重复名（同租户下）拒绝并提示"应用名已存在" |
| AC-102.5 | 应用创建有 audit log（actor、time、payload） |

### US-103：配置对象模型（"出差报销"对象）

**作为** 业务用户
**我希望** 在我的"出差报销"应用中创建一个叫"报销单"的对象
**并配置 5 个字段**：出发地、目的地、出差日期、金额、事由
**以便于** 我能用这个对象建表单和列表

#### 验收标准

```gherkin
Given 我在"出差报销"应用的对象管理页
When 我点击"新建对象" → 输入名称"报销单" → 添加字段：
  | 字段名   | 类型   | 是否必填 |
  | 出发地    | 文本   | 是       |
  | 目的地    | 文本   | 是       |
  | 出差日期 | 日期   | 是       |
  | 金额     | 数字   | 是       |
  | 事由     | 长文本 | 否       |
And 我点击保存
Then 数据库自动创建表 `app_reimbursement_<app_id>`
And 表里有 id, created_at, updated_at + 5 个字段
And 数据库 created_at 默认 NOW()
```

#### 详细 AC

| # | 条件 |
|---|------|
| AC-103.1 | 字段至少支持：文本 / 长文本 / 数字 / 日期 / 单选 / 多选（v1.0.1 MVP 范围） |
| AC-103.2 | 必填字段在保存表单时被强制校验 |
| AC-103.3 | 对象名在同应用内唯一 |
| AC-103.4 | 自动建表 SQL 经过 escape，SQL 注入测试通过 |
| AC-103.5 | 字段类型不可修改（要修改字段需先删除重建，并给出确认提示） |
| AC-103.6 | 对象模型有 ETag 版本号，保存时基于版本号做乐观锁 |

## 三、Epic M2：表单 + 列表（用户故事与 AC）

### US-201：表单编辑器

**作为** 业务用户
**我希望** 在"报销单"对象上点"创建表单"进入表单编辑器
**并通过拖拽把 5 个字段放到画布上**
**以便于** 我能定制业务表单

#### 验收标准

```gherkin
Given 我在"报销单"对象详情页
When 我点击"创建表单" → 系统默认创建一个名为"新建报销单"表单，进入编辑器
And 我从左侧控件库拖拽（出发地/目的地/出差日期/金额/事由）到画布
And 我点保存
Then 表单状态变更为"已发布"
And 数据库 app_form 表新增一条记录
```

#### 详细 AC

| # | 条件 |
|---|------|
| AC-201.1 | 编辑器左侧控件库列出本对象的可用字段 |
| AC-201.2 | 字段可拖拽到画布，并自动应用默认控件（文本→input、长文本→textarea、数字→number 等）|
| AC-201.3 | 每个控件支持"必填"开关 + 帮助文本 + 占位符 |
| AC-201.4 | 表单支持"草稿/已发布"两状态切换 |
| AC-201.5 | 表单布局：v1.0.1 仅支持 1 列布局（v1.0.2 起再加多列） |
| AC-201.6 | 画布支持撤销/重做（至少 5 步） |
| AC-201.7 | 预览按钮可立即在 iframe 内预览渲染效果 |

### US-202：渲染表单并提交

**作为** 终端用户
**我希望** 打开"新建报销单"表单页面，填写字段后提交
**以便于** 我的报销单能被记录

#### 验收标准

```gherkin
Given 我访问 /apps/<app_id>/forms/<form_id>/submit
And 我看到 5 个字段输入控件
When 我填写必填字段 + 提交
Then 数据库 app_form_data（已发布的表单数据存储表）插入一条记录
And 我看到"提交成功"toast
And 我被重定向到列表页
```

#### 详细 AC

| # | 条件 |
|---|------|
| AC-202.1 | 必填字段未填时，提交按钮无效 + 字段下方红字提示 |
| AC-202.2 | 字段类型校验：金额必须为数字、日期必须合法 |
| AC-202.3 | 富文本编辑器 v1.0.1 暂不实现 |
| AC-202.4 | 提交后保留已填值（避免重复输入） |
| AC-202.5 | 表单提交有 audit log |

### US-203：列表页 + 查询

**作为** 终端用户
**我希望** 在 /apps/<app_id>/forms/<form_id>/list 看到所有我提交过的报销单
**并能按"金额"降序查询**
**以便于** 我快速找到金额大的报销单

#### 验收标准

```gherkin
Given 我提交过 5 笔报销
When 我访问列表页 → 默认按创建时间倒序
And 我点击字段过滤"金额" → 输入 ">1000" → 点击应用
Then 列表显示金额 > 1000 的报销单
And 列表右上角显示结果数"3 条"
```

#### 详细 AC

| # | 条件 |
|---|------|
| AC-203.1 | 列表支持：分页（每页 20）、列排序（任意列）、列过滤（任意列） |
| AC-203.2 | 过滤支持：精确匹配 / 范围 / 模糊 / 是否为空 |
| AC-203.3 | 列头点击切换升/降序，再次点击取消排序 |
| AC-203.4 | 列表有"导出 CSV"按钮（v1.0.1 仅 CSV） |
| AC-203.5 | 列的可见性可配置（用户视角） |

## 四、Epic M3：简单流程（用户故事与 AC）

### US-301：流程编辑器（简化版）

**作为** 业务用户
**我希望** 在"新建报销单"表单上挂一个 2 节点审批流：
- 节点 1：申请人提交 → 直接到节点 2
- 节点 2：财务审批
**以便于** 大于 1000 的报销单需要财务审批

#### 验收标准

```gherkin
Given 我在表单详情页点击"配置流程"
When 我进入流程编辑器，拖出"开始 → 用户任务（财务角色）→ 结束"
And 我配置用户任务的"处理人"= 财务角色
And 我保存
Then 数据库 workflow 表新增一条记录，状态=已发布
```

#### 详细 AC

| # | 条件 |
|---|------|
| AC-301.1 | v1.0.1 流程节点仅支持：开始、用户任务、排他网关、结束 |
| AC-301.2 | 处理人支持 3 种：固定用户 / 角色 / 变量表达式 |
| AC-301.3 | 字段权限：财务可看金额，申请人不可看自己之外的报销 |
| AC-301.4 | 流程图存为 JSON（v1.0.1 不要求导出标准 BPMN 2.0，那是 v1.0.3） |
| AC-301.5 | 流程可启动/暂停/终止 |

---

### US-302：发起并审批流程

**作为** 申请人（demo）
**我希望** 提交一笔 2000 元的报销单
**然后** 在待办里看到我的报销单，并等财务审批

#### 验收标准

```gherkin
Given 我作为 demo 用户提交了一笔 2000 元的报销单
And 系统自动生成一条 workflow_instance
And 流程走到"财务审批"节点
When 我作为 demo-fin 用户登录（属于财务角色）
And 我进入"待办"页
Then 我看到这笔待办
And 我点击"通过" → 流程结束
And 我作为申请人能看到状态变更为"已通过"
```

#### 详细 AC

| # | 条件 |
|---|------|
| AC-302.1 | 发起流程时把表单数据快照存到 instance_payload |
| AC-302.2 | 待办实时刷新（轮询 5s，v1.0.1 不上 WebSocket） |
| AC-302.3 | 审批操作支持：通过 / 驳回 / 委派 / 加签（v1.0.1 仅通过 / 驳回） |
| AC-302.4 | 审批意见必填（驳回时），通过时可填可不填 |
| AC-302.5 | 流程实例详情页可视化展示节点流过的路径 |
| AC-302.6 | 申请人/审批人/超管 三类角色字段权限隔离 |

## 五、非功能性需求（NFR）

### 5.1 性能

| 项 | 指标 |
|----|------|
| 表单提交接口 P95 | < 500ms（v1.0.1）|
| 列表查询接口 P95 | < 800ms（数据量 1w 条内）|
| 表单编辑器首屏 | < 2s |
| 启动时间（含 5 个对象的 demo 应用） | 后端启动 < 10s，前端 < 3s |

### 5.2 兼容

- 浏览器：Chrome 100+ / Edge 100+ / Firefox 100+ / Safari 15+
- Node：18+
- Java：21+

### 5.3 安全

- 跨用户数据严格隔离：所有 API 必须在请求时校验 `app_id` 归属
- 表单提交需鉴权（不能裸 POST）
- SQL 注入测试通过（参数化查询）
- XSS 测试通过（自动转义）

### 5.4 可观测

- 后端关键接口有 console.log 记录请求 + 响应时间
- 错误响应统一格式：`{ code, message, traceId }`

|----|------|-------------------|

## 七、Demo 剧本（验收会用）

> 这是评审会用操作脚本，10 分钟跑完。

### 步骤

1. **登录**：访问 http://localhost:5173/login → demo / demo1234
2. **进入应用中心**：自动跳到 /apps，看到"订单管理"种子应用
3. **新建应用**：点击"新建应用"→ 名 "出差报销" → 选 icon → 提交
4. **新建对象**："报销单"对象 → 加 5 字段：出发地（文本 必填）、目的地（文本 必填）、出差日期（日期 必填）、金额（数字 必填）、事由（长文本）
5. **建表单**：点"创建表单"→ 拖拽 5 字段 → 保存为"新建报销单"
6. **建流程**：在表单上点"配置流程"→ 拖出"开始 → 用户任务（财务角色）→ 结束"→ 保存
7. **发布**：表单状态切到"已发布"
8. **切换申请人提交**：登出，登录 demo2 → 进入表单 → 填 2000 元 → 提交
9. **切换财务审批**：登出，登录 demo-fin → 进入"待办"→ 看到报销单 → 点"通过"
10. **回到申请人**：登出，登录 demo2 → 进入"我的申请"→ 看到状态"已通过"

**评审用关键 KPI**：
- 全程 ≤ 10 分钟
- 0 个未捕获错误
- 所有 API 响应 P95 < 500ms

## 八、风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| ontologoy-engine 自动建表的 SQL 兼容性 | 高 | Sprint 0 先验证手动建表脚本回放 |
| 表单编辑器的撤销/重做状态机复杂度 | 中 | 借鉴 redux undo 中间件，最少改动实现 |
| 流程引擎的乐观锁导致的死锁 | 中 | 每次拿实例先加 200ms 超时；冲突回滚提示 |
| 6 类种子用户/角色的密码管理 | 低 | demo 阶段统一 demo1234 |
| 同时多人编辑同一表单 | 中 | v1.0.1 暂不支持，锁机制推到 v1.0.2 |

- ❌ 多列表单布局、卡片式表单 → v1.0.2

- ❌ WebSocket 实时通知 → v1.0.5
- ❌ SSO/ABAC/字段脱敏 → v1.0.5
