MetaPlatform — "积木"的颗粒度

您说"像搭积木一样构建业务系统"。问题是：积木的最小单元是什么？不同选择决定了客户的心智模型和平台的扩展性。

G1 · UI 单元 (表单/页面/按钮)

积木就是"UI 组件 + 事件"。对应 Retool / 飞书多维表格。上手快、所见即所得；但每个"积木"都和具体业务耦合，复用性差，AI 难以跨应用推理。

G2 · 业务对象 (Entity-First)

积木是"客户/订单/合同"这样的业务对象，UI/流程/权限都是"派生"。对应 Palantir Ontology。业务侧心智模型最自然；AI 推理质量最高；客户必须先"建模"。

G3 · 能力 (Capability-First)

积木是"做一件事的完整能力"：发邮件、跑 OCR、开审批、做报表。每个能力有标准输入/输出/配置/版本，跨应用、跨业务对象复用。对应"微服务 / Serverless 函数市场"。

G4 · 场景 (Scenario-First)

积木是"采购到付款"这种端到端业务场景。AI Agent 是主要使用者，平台提供"场景模板 + 积木库"。最像"产品"而不像"工具"；研发成本最高、商业想象空间最大。

小提示

这四档不是互斥的——一个平台通常会同时存在多种颗粒度的"积木"。问题在于主积木是哪种。
