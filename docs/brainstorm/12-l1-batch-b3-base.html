MetaPlatform · L1 细化 · B3 底座四层（直接文字版 · 浏览器伴侣版）

B1（用户面）+ B2（核心）+ B3（底座）= 完整架构。B3 是企业级入场券，行业成熟方案多，概念创新少，重在选型。

B3-1 · AI Substrate (贯穿全栈的能力基质) v2 新增

v2 关键变化：v1 中"AI 胶水层"和"能力库"是两层；v2 中AI 胶水层升级为"AI Substrate"——是渗透在每两层之间的能力基质，不再是"一层"，而是一组横切能力。

职责：为 MetaPlatform 所有层提供 AI 能力。不直接面向用户/Agent；被所有层调用。

关键概念：

  • LLM Gateway — 多模型路由；按"成本/速度/质量/隐私"自动选模型（OpenAI / Claude / DeepSeek / Qwen / 自托管）
  • Embedding Service — 统一向量化（文本/图片/音视频）；可对接多家模型
  • Vector Store Adapter — 抽象向量库（Milvus / Qdrant / pgvector / Elasticsearch）
  • Context Store — 跨会话/跨页面/跨用户的"上下文记忆"；按"会话/工作面/企业/全局"四级隔离
  • Agent Runtime — Agent 编排引擎（计划/反思/工具调用/人机协作）；支持自定义 Agent
  • Prompt Registry — 提示词版本管理 + A/B 测试 + 效果评估
  • Token 计费 — 按模型/调用次数/Token 量的计费中心；企业内/企业间双层计费
  • 模型微调平台 — 用客户业务数据微调行业模型（私有部署）

AI 贯穿体现：AI Substrate 不在架构图中显示为"层"——它渗透在所有层中。B1 v2 中所有"AI 嵌入式控件"、B2 中所有"AI 字段/NL 建模/图谱 RAG"都来自 AI Substrate。

上下游：

  • 向上：被 B1 各层 + B2 各层 + 平台底座的 AI 增强模块 调用
  • 向下：模型 API（外部）/ GPU 集群（自建）/ 内部模型网关

典型数据流：用户在对话层说"做客户列表" → 嵌入式 NL 解析器（AI Substrate 的 Intent Parser）→ 调用 LLM Gateway → 模型返回 JSON 描述 → 解析为业务对象配置 → 触发业务对象层创建

B3-2 · 能力库 (G3 横切原子能力)

职责：把"做一件事的完整能力"封装成可复用积木。横切于所有业务对象、所有流程。

关键概念：

  • 能力 (Capability) — 一段可调用的"做事"代码 + 元数据
  • 输入/输出 Schema — 能力的标准接口；JSON Schema 描述
  • 配置 (Configuration) — 调用时传入的参数（API Key / 模板 / 阈值）
  • 版本 (Version) — semver 语义版本；可灰度/回滚
  • SLA — 响应时间 / 可用性 / 重试策略
  • 可观测性 — 每次调用的日志/指标/链路追踪
  • 🆕 AI 能力标记 (AI Tag) — v2 新增：能力可以是"AI 能力"（如"AI 翻译"）或"传统能力"（如"发送邮件"）

典型能力清单（MVP 至少 10 个）：

能力 类型 用途
邮件发送传统SMTP/企业微信/钉钉
短信发送传统阿里云/腾讯云
OCR 识别AI提取图片/PDF 中的文字
AI 翻译AI跨语言
AI 摘要AI长文本/音视频/合同
PDF 生成传统合同/发票生成
数字签名传统合同签署
鉴权传统OAuth/SAML/OIDC
数据同步传统跨业务对象数据搬运
计费传统触发计费事件
🆕 NL→业务对象操作AI解析自然语言为 CRUD
🆕 AI 推荐AI"下一步建议"
🆕 流程挖掘AI流程效率分析

典型数据流：流程节点"客户提交后通知销售" → 调用能力"邮件发送" → 配置收件人+模板+变量 → 执行 → 记录调用日志/指标/错误重试

B3-3 · 平台底座（多租户 / 权限 / 审计 / 集成 / 治理）

职责：让 MetaPlatform 能"进企业"——多组织隔离、权限控制、审计合规、外部系统接入、数据治理。

关键概念：

  • 租户 (Tenant) — 一家企业；可含子组织/部门/团队
  • 用户 (User) / 角色 (Role) / 权限 (Permission) — RBAC + ABAC 混合
  • 行级 / 列级 / 字段级权限 — 谁能看哪些业务对象实例/字段
  • 审计 (Audit) — 所有操作留痕（谁/何时/做了什么）；可对接企业 SIEM
  • 数据血缘 (Lineage) — 数据从哪里来到哪里去
  • 数据质量 (Quality) — 完整性/准确性/一致性/及时性
  • 元数据 (Metadata) — 业务元数据/技术元数据/管理元数据
  • Integration Hub — 外部系统接入中心
    • 连接器 (Connector) — ERP (SAP/Oracle) / CRM (Salesforce/纷享销客) / DB (MySQL/Oracle/PG) / MQ (Kafka/RabbitMQ) / API (REST/GraphQL/Webhook) / 文件 (SFTP/CSV/Excel)
    • 映射 (Mapping) — 外部数据 → 平台业务对象
    • 同步策略 — 实时/定时/手动/事件驱动
    • 冲突解决 — 同一数据多源时的优先级
  • 🆕 AI 集成增强 (v2) — AI 推断数据源关系 / AI 自动生成映射 / AI 数据质量评分

上下游：

  • 向上：所有层（业务对象层/本体引擎/数据知识层/AI Substrate）都受平台底座的"管控"
  • 向下：存储底座（账号库/审计库）+ 外部系统（通过 Integration Hub）

典型数据流（新员工入职）：HR 在 SAP 中创建员工 → SAP 触发 Webhook → Integration Hub 接收 → 映射为"员工"业务对象 → 写入本体引擎 → 自动创建默认视图 + 流程模板（入职清单）→ AI Substrate 推断"需要给该员工分配哪些应用权限"→ 平台底座自动配置

B3-4 · 存储与基础设施

职责：底层数据持久化与基础设施。

关键选型（MVP 建议）：

组件 选型 用途 备注
关系数据库PostgreSQL 16+业务元数据/审计/租户/权限唯一真源必须用 PG
图数据库Neo4j 5+本体图谱 / 实体关系P3 架构核心
向量数据库Milvus 2.4+文档/多模态 EmbeddingRAG/图谱 RAG
对象存储MinIO / S3 兼容文件/音视频/备份信创要求
缓存Redis 7+会话/热点/分布式锁-
搜索引擎Elasticsearch 8+全文检索/日志可选
消息队列Kafka / NATS事件总线/异步P3 双向同步依赖
时序库（可选）InfluxDB / TimescaleDB监控指标/审计v2 起

信创/国产化适配（如有要求）：替换为 OceanBase/TiDB/达梦/GuassDB；图库可考虑 HugeGraph；向量库可替换为 Milvus 信创版。

B3-5 · 部署与交付

职责：让 MetaPlatform 能在任何环境跑起来。

关键概念：

  • 容器化 — Docker + Kubernetes（Helm Chart）
  • 多云适配 — AWS/Azure/阿里云/腾讯云/华为云
  • 私有化部署 — 离线包/一键安装
  • 边缘部署 — 边缘节点（工厂/门店）
  • 🆕 信创适配 (v2 重要) — 国产 CPU（鲲鹏/飞腾/龙芯）+ 国产 OS（统信/麒麟/欧拉）
  • 灰度发布 — 按租户/按地域/按用户群
  • 备份与恢复 — 跨地域/跨云/快照/增量
  • 可观测性 — Prometheus + Grafana + Loki + Tempo

典型场景：

  • SaaS 模式 — MetaPlatform 托管；客户按席位/按用量付费
  • 专有云模式 — 部署在客户云上；客户拥有数据
  • 私有化模式 — 部署在客户机房；离线/气隙环境
  • 边缘模式 — 工厂车间；数据不出工厂
B3 小结
层 一句话总结 关键决策
AI Substrate贯穿全栈的 AI 能力基质v2 新增；替换原"AI 胶水层"
能力库 (G3)10+ 横切原子能力；AI 能力可标记MVP 必含
平台底座多租户+权限+审计+Integration Hub+治理企业级入场券
存储底座PG/Neo4j/Milvus/MinIO/Redis/Kafka选型稳定
部署交付K8s+Helm+多云+私有化+信创信创是大型企业门槛

B3 整体特点：行业成熟方案多，新概念创新少（B1/B2 才是灵魂）；选型决策比概念决策更关键；国产化/信创是"是否能进大型企业"的硬门槛。
