## 1. 产品愿景与定位

### 1.1 一句话

> **MetaPlatform 是一个基于本体论（Ontology）的超级业务平台。它把企业的业务建模、应用开发、流程自动化、知识管理、数据治理整合到一个统一平台上，让企业像搭积木一样构建和运营业务系统，而 AI 和本体论是积木之间的"胶水"。**

### 1.2 产品定位

| 维度 | 定位 | 关键含义 |
|---|---|---|
| **市场定位** | "低代码 + 知识图谱"型平台（B） | 卖业务交付能力，不是底层 OS；介于 Palantir Foundry（基础设施型）与 OutSystems（应用型）之间 |
| **目标客户** | 做数字化 / 智能化转型的企业 | 业务部门 + 转型办联合驱动；中大型企业为主 |
| **核心差异化** | "本体论 = 单一真源" + "AI 贯穿全栈" | 与传统低代码的"低代码 + 表单"拉开代差；与 Palantir 的"本体优先"对齐 |
| **商业化路径** | SaaS（按席位/用量）+ 专有云 + 私有化 + 边缘 | 多模式部署；信创是大型企业门槛 |

### 1.3 明确不做（Non-Goals）

- ❌ **不做底层操作系统**——不是 Palantir 路线（不卖数字基础设施）
- ❌ **不做 AI-Only Agent**——不是 Glean 路线（AI 是基质，不是唯一用户）
- ❌ **不做单一行业垂直 SaaS**——保持通用 + 行业模板的扩展模式
- ❌ **不做"低代码装饰本体论"**——本体论是灵魂，不是事后装饰

### 1.4 明确做（Goals · v1.1 反转补充）

> 🆕 v1.1（用户最新指示）：**"数据栈是平台能力。数据湖、数据仓库都需要做"**。
> 这是对原 D6 "数据湖/数仓只对接" 的反转。

- ✅ **数据栈 = MetaPlatform 平台能力的组成部分**——不是外部依赖
- ✅ **数据湖**（Hudi/Iceberg/Schema 演进）平台**自建存储 + 接入外部**
- ✅ **数据仓库**（OLAP/指标/维度/仪表盘）平台**自建引擎 + 接入外部数仓**
- ✅ **不做的事**：不取代 Snowflake / Databricks 这种"通用超大规模数仓"——平台自带数仓针对**业务指标 + 中等数据量**场景，足以覆盖 80% 企业客户；超大客户用适配器接外部数仓

**这条反转对全 spec 的影响**：
- 第 4.3、5.2.3、7、10.2、12.1、14.1 中所有"半对接/弱对接"的提法都升级为"内置 + 适配器"
- 新增子项目：**数据湖 v0.1**、**数据仓库 v0.1**（均升为 🔴 P0）

## 2. 目标客户与场景

### 2.1 客户画像

| 客户类型 | 典型规模 | 主要痛点 | MetaPlatform 给的价值 |
|---|---|---|---|
| **数字化转型办** | 中大型企业（员工 1k+） | 业务部门提需求慢、IT 响应慢、系统烟囱林立 | 一个平台承接所有需求；本体论统一语义 |
| **业务部门负责人** | 各业务线 | 想要"自己的系统"但 IT 排期 1 年+ | 业务人员 + IT 协作建模，1-2 周跑通第一个应用 |
| **数据/AI 团队** | 数据智能部门 | 数据治理难、AI 落地难 | 本体 + AI 贯穿 + 数据统一访问 |
| **IT 架构师** | 企业 IT | 系统多、技术债重、新技术引入慢 | 一个平台替代 5-10 个工具 |

### 2.2 典型场景

1. **新建业务系统**（替代传统开发）——业务建模 → 页面/流程自动派生 → 上线运营
2. **改造遗留系统**（Integration Hub 接入）——老系统数据接入平台 → 在平台上做新功能
3. **统一知识管理**（RAG + 知识图谱）——文档/合同/客户关系一站式管理
4. **AI 改造核心流程**（AI 节点 + 智能派单）——AI 嵌入流程决策、文档处理、客户服务
5. **跨部门数据治理**（MDM + 数据血缘）——主数据统一管理；数据来源可追溯

### 2.3 不针对的客户

- ❌ **创业小团队**——这类客户更适用 SaaS 工具（如飞书多维表格 + 飞书捷径）
- ❌ **已经深度绑定 SAP/Oracle**——切换成本太高
- ❌ **纯 C 端产品公司**——MetaPlatform 是 B 端平台

## 3. 核心架构原则

### 3.1 三大架构原则

#### 原则 1：本体论 = 单一真源 (Single Source of Truth)

业务对象 / 流程 / 视图 / 权限 / AI 知识 / 数据血缘，**所有业务语义都源于本体**。本体引擎是 P3 双向同步的"心脏"。

#### 原则 2：AI 贯穿全栈 (AI as Substrate)

**AI 不是"一层"，是渗透在每两层之间的能力基质**。每个组件（表单字段、列表列、流程节点）都自带 AI 嵌入式增强。这一原则源自用户原话："AI 和本体论是积木之间的胶水"——胶水不只在一层，在每两层之间。

**v1 → v2 关键修正**：
- v1 把"AI 胶水层"作为独立紫色层
- v2 升级为"AI Substrate"——是一组横切能力，被所有层调用

#### 原则 3：平台自举 (Platform Bootstrapping)

**平台前端 = 自己用低代码搭；平台核心能力 = 代码写**。这是用户明确指出的自举原则。

| 类别 | 自举范围 | 不自举范围 |
|---|---|---|
| 平台前端 UI | 建模特工场、租户管理、计费后台、审计可视化、工单系统、运营 Dashboard、应用市场前端 | — |
| 平台核心能力 | — | 本体引擎、AI Substrate、存储引擎适配、Integration Hub 内核、安全框架、部署 K8s Operator |

**自举带来的产品效应**：
- ✅ 平台升级时，自身 UI 跟着升级（自我演化）
- ✅ 客户看到的"平台界面"和"用平台搭的应用界面"风格完全一致
- ✅ 平台的内置流程（计费/审计/工单）是"业务对象"——可被客户复用

### 3.2 次级原则

- **4 个不变原则 (Invariants)**：① 本体论是单一真源；② AI 贯穿；③ 自举前端；④ 三角咬合（业务对象 ↔ 本体 ↔ 数据/知识）
- **演进性**：MVP 阶段不强求"完整双向同步"，用**半自动双向**作为务实实现
- **可逆性**：所有客户业务对象定义必须支持版本回滚；本体变更必须可回滚
- **可观测性**：所有操作留痕；本体变更可追溯到"谁/何时/为什么"

## 4. 顶层架构总览

### 4.1 V2 架构分层（9 层 + 1 基质）

```
┌─────────────────────────────────────────────────────────┐
│  L1-1 用户与 AI 对话层 (NL UI / 拖拽 / 移动 / Agent)   │  ← 蓝色（用户面）
│  L1-2 应用与场景市场 (G4 二期)                          │
│  L1-3 页面与表单生成器 (G1 派生)                        │
│  L1-4 流程自动化引擎 (BPMN / 状态机 / 规则)            │
├─────────────────────────────────────────────────────────┤
│  L2-1 业务对象层 (G2 主积木) — MVP 核心                │  ← 橙色（灵魂）
│  L2-2 本体引擎 + 知识图谱 (P3 双向同步)                │  ← 绿色
│  L2-3 数据/知识统一层 (RAG/非结构/MDM/湖/仓)          │  ← 红色
├─────────────────────────────────────────────────────────┤
│  L3-1 AI Substrate (贯穿全栈的能力基质)                │  ← 紫色（基质）
│  L3-2 能力库 G3 (10+ 横切原子能力)                     │
│  L3-3 平台底座 (多租户/权限/审计/Integration/治理)    │  ← 灰色
│  L3-4 存储与基础设施 (PG/Neo4j/Milvus/MinIO/Kafka)    │
│  L3-5 部署与交付 (K8s+Helm+多云+信创)                 │  ← 黑色
└─────────────────────────────────────────────────────────┘

🟣 AI Substrate 渗透在所有层之间（不是层，是基质）
```

### 4.2 关键耦合关系（灵魂三层三角咬合）

| 关系 | 方向 | 机制 |
|---|---|---|
| 业务对象 ↔ 本体 | 双向 | P3 半自动同步；变化需建模者确认 |
| 业务对象 → 数据/知识 | 向下 | 实例数据持久化到 RAG/MDM/湖/仓 |
| 本体 → 数据/知识 | 向下 | 图谱数据存 Neo4j + 关系引用 |
| 数据/知识 → AI Substrate | 向上 | AI 消费 5 类数据（图谱 RAG / 文档 RAG / NL2SQL / MDM / 湖） |
| AI Substrate → 业务对象/本体 | 向下 | AI 建议进入"待确认"区，建模者点确认才写入 |

### 4.3 数据/知识策略（v1.1 反转）

| 子层 | 模式 | 含义 |
|---|---|---|
| RAG 知识库 | 🟥 **内置** | 平台自带全文 + 向量 + 文档版本/协同 |
| 非结构化知识库 | 🟥 **内置** | 平台自带文件/音视频/PDF 的存储、解析、OCR、版本、权限 |
| 主数据 (MDM) | 🟥 **内置** | 平台自带黄金记录管理、去重、合并、治理 |
| 数据湖 | 🟥 **内置 + 适配器**（v1.1 反转）| 平台自带 Hudi/Iceberg 存储引擎；同时通过适配器消费外部湖（Hudi/Iceberg/Delta/MinIO） |
| 数据仓库 | 🟥 **内置 + 适配器**（v1.1 反转；v1.2 D13 明确）| 平台自带 **Apache Doris** 引擎；同时通过 SQL 适配器消费外部数仓（ClickHouse/Snowflake/BigQuery） |

**核心承诺**（v1.1）：MetaPlatform **自带数据栈 + 统一访问、统一语义**。
- 自带数据栈针对**业务指标 + 中等数据量**场景（足以覆盖 80% 企业客户）
- 超大规模数仓（Snowflake/Databricks）通过适配器接入，平台**不取代**通用超大数据基础设施
- 数据栈的"自建 + 接入"双轨模式，是 v1.1 对 v1.0 "只对接"决策的关键升级

## 5. L1 细化：9 层设计

### 5.1 B1 用户面三层

#### L1-1 · 用户与 AI 对话层 🆕 AI 贯穿

**职责**：所有用户触点的统一入口。承担 5 类交互形态：自然语言对话、Web 拖拽、移动端原生、AI Agent API、企业 ChatOps 嵌入。

**关键概念**：会话 (Session) / 工作面 (Workspace) / AI 助手 (Copilot) / 指令 (Intent) / 回复 (Render) / **🆕 AI 嵌入式控件** / **🆕 上下文记忆 (Context Memory)** / **🆕 NL 解析器 (嵌入式)** / **🆕 多模态输入**

**上下游**：向上接所有用户/Agent/SDK；向下接业务对象层 + 流程自动化引擎

**典型数据流 (v2 增强版)**：
- **NL 操作流**：用户输入"把所有北京的客户标为 VIP" → 嵌入式 NL 解析器 → 解析为业务对象操作 → 调用业务对象层
- **🆕 多模态流**：用户拍一张名片 → AI OCR + NER 提取人名/电话/公司 → 建议"是否创建客户对象？" → 自动填表
- **🆕 智能补全流**：用户在"客户名称"输入"阿里" → AI 实时建议"阿里巴巴西溪园区"
- **🆕 跨会话流**：用户上次说"我关心上海客户"，本次打开直接看到"上海客户看板"

#### L1-2 · 应用与场景市场 (G4 二期)

**职责**：把"业务对象 + 流程 + 视图"组合成"可销售/可复用的端到端应用包"。

**关键概念**：应用包 / 场景模板 / 发布（四级可见性）/ 订阅 / 差异 (Diff) / **🆕 AI 语义搜索** / **🆕 AI 业务匹配度评分** / **🆕 AI 升级合并助手** / **🆕 AI 应用质量评估**

**典型数据流 (v2)**：用户说"我需要一个采购到付款的模板" → AI 检索市场 → 推荐 3 个 → AI 评估"匹配度 82%，建议补充 2 个业务对象" → 一键安装

#### L1-3 · 页面与表单生成器 (G1 派生) 🆕 AI 贯穿

**职责**：把"业务对象定义"自动 + 半自动地转成 UI。**P3 架构中"从本体派生 UI"的具体执行层**。

**关键概念**：视图 / 表单 / 列表 / 看板 / 组件 / 布局 / 权限投影 / **🆕 NL 页面生成器** / **🆕 智能布局** / **🆕 字段语义识别** / **🆕 AI 异常 UI 检测** / **🆕 NL 微调**

**典型数据流 (v2)**：用户说"做一个客户列表，显示近 30 天活跃度" → AI 解析 → 自动配置列表字段 + 过滤 + 排序 → 预览

#### L1-4 · 流程自动化引擎 🆕 AI 贯穿

**职责**：让业务跑起来。支持 BPMN/状态机/规则三种范式并存。

**关键概念**：流程定义 / 流程实例 / 触发器 / 节点 / SLA / 参与者解析 / 补偿 / **🆕 NL 流程设计** / **🆕 AI 节点（AI 决策节点）** / **🆕 流程挖掘** / **🆕 异常检测** / **🆕 AI 智能派单** / **🆕 NL 流程解释**

**典型数据流 (v2)**：用户提交"采购申请" → 触发流程 → AI 节点"金额异常检测"判断"该申请人近 30 天有 3 次类似申请" → 标注"建议人工复审" → 经理收到带 AI 建议的审批任务

### 5.2 B2 核心三层（灵魂层）

#### L2-1 · 业务对象层 (G2 主积木) 🆕 AI 贯穿

**职责**：MetaPlatform 的"业务语义中心"。所有客户业务对象的定义、实例、关系、规则都在这一层。

**9 个关键概念**：对象类型 / 对象实例 / 字段 / 关系 / 业务规则 / 视图配置 / 生命周期 / 数据来源 / 权限

**5 个 AI 贯穿新增**：🆕 AI 字段类型（AI 提取/生成/分类字段）/ 🆕 NL 建模 / 🆕 AI 关系建议 / 🆕 AI 规则发现 / 🆕 AI 字段补全

**典型数据流**：建模者 NL 输入"做一个'客户'对象" → AI 建议字段 → 建模者确认 → AI 映射到本体 → 自动生成 UI → 用户填数据 → 业务规则生效 → 持久化

#### L2-2 · 本体引擎 + 知识图谱 (P3 双向同步核心) 🆕 AI 贯穿

**职责**：语义枢纽和知识图谱运行时。实体模型 / 关系图谱 / 推理 / 语义查询 / 事件总线。**P3 架构中"单一真源"的具体实现**。

**9 个关键概念**：实体类型 / 属性 / 关系 / 公理 / 推理 / 语义查询 / 图算法 / **事件总线** / **版本 (Versioning)**

**5 个 AI 贯穿新增**：🆕 NL 本体编辑 / 🆕 AI 本体检查 (Ontology Linter) / 🆕 AI 推理建议 / 🆕 AI 异常图谱检测 / 🆕 **图谱 RAG (GraphRAG)**

**P3 双向同步典型数据流**：
1. 建模者创建"供应商"对象 → 本体引擎接收 → 写入 Neo4j（图）+ PG（元数据）
2. 事件总线广播"EntityCreated: Supplier" → 页面生成器自动生成 UI
3. 用户添加"供应商评级"字段 → 业务对象层变化 → 触发"半自动双向同步" → 弹出"确认要将此改动写入本体？" → 建模者确认 → 本体更新
4. AI Substrate 收到事件 → 推断"是否生成 AI 评估规则？" → 提示建模者

#### L2-3 · 数据/知识统一层 (5 类) 🆕 AI 贯穿

| 子层 | 模式 | 关键概念 | AI 增强 |
|---|---|---|---|
| 3.1 RAG 知识库 | 🟥 内置 | 文档库/索引/向量/检索/重排/引用 | NL 文档问答 / 自动 FAQ / 文档版本 / 跨文档推理 |
| 3.2 非结构化知识库 | 🟥 内置 | 对象存储/版本/权限/标签/OCR/转码 | AI 打标签 / 摘要 / 分类 / 跨文件关联 |
| 3.3 主数据 MDM | 🟥 内置 | 黄金记录/匹配/合并/生存/质量 | AI 匹配 / 合并规则建议 / 质量评分 / 数据血缘 |
| 3.4 数据湖 | 🟥 内置 + 适配器（v1.1）| 平台自带 Hudi/Iceberg 存储引擎 + Schema 演进；外部湖通过适配器接入 | AI Schema 推断 / 数据探索 / 跨源 JOIN 建议 |
| 3.5 数据仓库 | 🟥 内置 + 适配器（v1.1；D13 明确 Doris）| 平台自带 **Apache Doris** 引擎 + OLAP Cube/指标/维度/仪表盘；外部数仓通过 SQL 适配器 | AI 指标推荐 / 异常检测 / NL→SQL |

## 7. 数据/知识策略（v1.1 反转）

详见第 4.3 节。**v1.1 核心承诺**：**MetaPlatform 自带数据栈**——数据湖/数据仓库是平台能力的组成部分，**不再是"对接"**。

**v1.0 → v1.1 反转说明**：
- v1.0："暂不深入" = 平台**自己不重做**数仓，而是**对接**外部数仓
- v1.1：用户明确指示"数据栈是平台能力。数据湖、数据仓库都需要做"
- 平台自带的数仓定位 = **业务指标 + 中等数据量**，覆盖 80% 客户
- 超大规模数仓（Snowflake/Databricks）走适配器

**5 类数据/知识能力的内建比例**（v1.1）：

| 能力 | 内置 | 适配器 | 含义 |

| RAG 知识库 | 100% | 接入外部 Embedding 模型 | 全栈内置 |
| 非结构化知识库 | 100% | 接入外部 OCR 模型 | 全栈内置 |
| 主数据 MDM | 100% | 接入外部数据源 | 全栈内置 |
| 数据湖 | 自带 Hudi/Iceberg | 接外部 Hudi/Iceberg/Delta | **v1.1 反转** |

## 9. 平台自举原则

### 9.1 自举清单（自举 vs 不自举）

详见第 3.1 节原则 3 的表格。

### 9.2 自举的关键产品效应

1. ✅ **Eat your own dog food** — 平台自己用，bug 第一手发现
2. ✅ **狗粮即 demo** — 客户看到"我们连自己都用这个搭"，信任度↑
3. ✅ **配置变更可审计** — 低代码的修改自动留痕
4. ✅ **需求驱动平台进化** — 平台自身的不便变成"必须做"的优化
5. ✅ **风格完全一致** — 平台 UI 和应用 UI 同源
6. ✅ **可复用** — 客户可以照搬"工单业务对象"做出自己的工单系统

### 9.3 自举的边界规则

- ✅ 用业务对象 + 流程 + 表单 + 视图 + 能力调用能搭的 → 自举
- ❌ 需要"性能极致" / "安全/加密" / "硬件/驱动集成" / "复杂图算法" → 代码
- ⚠️ 模糊地带 → 默认自举（让平台自身用，bug 第一手发现）

### 9.4 关键洞察

> **MetaPlatform 的"建模特工场"本身就是用 MetaPlatform 搭出来的。**  
> 客户第一次登录看到的"建模特工场"界面 = MetaPlatform 业务对象"建模器配置"的可视化。  
> **这意味着**：客户也可以基于"建模器配置"业务对象，**搭建自己的"业务对象建模器"**——比如做一个"行业版建模器"（如"医疗对象建模器"）作为差异化产品。

这是 MetaPlatform **长期生态**的关键设计。

## 10. MVP 范围与边界

### 10.1 MVP 目标（v1.2 修订 · 1+6+3 月分期）

> 🆕 **v1.2 用户决策（D14）**：MVP 不再是"6-9 个月一气呵成"，而是 **1 + 6 + 3 月三期**。
>
> - **第 1 期 · 1 月（T-1 ~ T0）：核心可行性验证（Spike）**
>   - 目标：用 **最小可运行骨架** 验证 P3 本体引擎 + 数据栈能跑通
>   - 范围：本体引擎 v0.1（Neo4j 单租户）+ 数据栈 v0.1（Hudi 湖 + Doris 仓）最小化
>   - 验收：能在控制台建一个"客户"实体 → 写入图 + 湖 → 用 SQL 查出来
>   - 这是个**研究验证阶段**，不是产品阶段；为后续决策提供"实物证据"
>
> - **第 2 期 · 6 月（T0 ~ T+6m）：MVP 完整功能**
>   - 目标：可演示给第一个付费客户的核心场景
>   - 范围：B1 全部 + B2 全部 + B3 底座核心 + 数据栈 P0（数据湖 + 数仓）
>   - 验收：在企业内部能用 MetaPlatform 完成"业务建模 → 应用生成 → 数据治理 → AI 增强"全流程闭环
>
> - **第 3 期 · 3 月（T+6m ~ T+9m）：MVP 加固 + 商用化前置**
>   - 目标：补全 D6 数据栈 + 信创适配 + 安全合规 + 性能压测
>   - 范围：Doris 性能调优、Hudi/Iceberg 双栈、ClickHouse 适配器、安全加固、SLA 文档
>   - 验收：可签首单 SaaS / 专有云合同
>
> **累计**：T-1 ~ T+9m = **10 个月总周期**（与 v1.1 "6-9 月" 相比延长 1 个月，多 1 月验证）

> v1.0 时期建议的"本体引擎 v0.1 单兵突进"在 v1.2 升级为**双线并行**：

- D6 反转后，业务对象层/AI Substrate/平台底座都依赖数据栈的持久化能力

- 第 1 期 Spike 阶段会做**集成验证**（写一个跨图+湖的最小 e2e），证明两条线能合到一起

### 15.2 术语表

| 术语 | 定义 |
|---|---|
| **本体 (Ontology)** | 业务对象的形式化定义，含实体/属性/关系/公理 |
| **本体引擎** | 管理本体的运行时，MetaPlatform 的"单一真源" |
| **业务对象** | 客户视角的"实体"；与本体一一对应 |
| **半自动双向同步** | P3 架构的务实实现：任何"业务对象变化"进入"待确认"区；建模者点确认才写入本体 |
| **AI Substrate** | v2 概念；AI 不是一层，是渗透在所有层之间的能力基质 |
| **G2 / G3 / G4** | 主积木 / 横切能力 / 场景模板（按 B1 v2 决策） |
| **P3** | 双向同步架构（按 D3 决策）|
| **MVP** | Minimum Viable Product；本 spec 中为 **1+6+3 月分期**（v1.2 D14 决策）|
| **半自动双向** | P3 双向同步的 MVP 实现；人工审批降级为工作流问题 |
| **自举 (Bootstrapping)** | 平台前端 = 平台自身用低代码搭建 |
| **图谱 RAG (GraphRAG)** | AI 不只查文档，还查图谱关系 |
