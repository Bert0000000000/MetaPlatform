# 阶段二真实验收与第三阶段并行推进设计

> 状态：已确认，作为执行约束
>
> 日期：2026-07-17

## 目标

在不重复开发阶段二已完成代码的前提下，完成本地真实跨服务验收；同时从 TECH-A2A 开始第三阶段实现。阶段二里程碑只有在真实依赖链路验证通过后，才从 `[~]` 更新为 `[x]`。

## 验收范围

阶段二验收覆盖以下 7 个里程碑：

1. Agent 运行时：TECH-AGENT → TECH-RAG → TECH-ACTION → TECH-LLMGW，验证 trace_id、执行步骤和多步结果。
2. 数字员工 MVP：APP-DW → TECH-AGENT/RAG/ONT，验证创建、能力绑定、文档上传、抽取、审核写入和对话。
3. SuperAI 全模式：APP-SUPERAI → LLMGW/RAG/ONT/ACTION，验证问答、分析、Action、Ontology 探索、编排和调度入口。
4. 流程审批：APP-APPHUB → TECH-WFE，验证流程设计、发布、实例、审批和事件。
5. MCP 协议：外部 MCP Client → TECH-MCP → ONT/RAG/ACTION，验证发现、调用、批量和异步执行。
6. RAG 增强：TECH-RAG → ONT/图谱，验证图谱增强、上下文组装、引用溯源和质量指标记录。
7. LLMGW 管控：真实模型请求 → Prompt/配额/限流，验证调用、配额消耗、限流统计和成本数据。

## 验收环境

- 基础设施使用 `infra/docker-compose.base.yml`，包括 PostgreSQL、Redis、Kafka、Neo4j、Loki、Prometheus 和 Grafana。
- 业务服务按各模块现有启动方式分别启动，不修改既有 API 契约。
- LLM 使用本地已配置的真实模型凭证；凭证只通过环境变量注入，不写入仓库、不输出到日志。
- 验收数据使用独立测试租户和带有明确前缀的测试对象，避免污染已有数据。

## 验收记录

每个里程碑记录：

- 前置服务和健康检查结果。
- 请求路径、关键请求参数摘要、响应状态和业务结果。
- `trace_id` 是否贯穿调用链。
- 失败服务、错误码、日志位置和复现命令。
- 是否满足路线图验收标准。

单元测试、模块构建和 Mock Client 结果只能作为代码级证据，不能替代真实 E2E。

## 第三阶段并行范围

第一批只推进 TECH-A2A，不同时改动 APP-DW、APP-SUPERAI 或其他 P3 模块，避免协议契约尚未稳定时产生跨模块返工。

TECH-A2A 按以下依赖顺序推进：

1. Agent Card 发布/查询/公开端点。
2. Agent 注册、发现和健康检查。
3. 出站任务委托、状态查询、结果获取和取消。
4. 入站任务接收与 TECH-AGENT/TECH-WFE 路由。
5. SSE 进度、认证和 Outbox 事件。
6. 消息管理、审计统计、API Key 和任务高级管理。

每个小批次必须先补回归测试，再实现最小代码，最后运行模块测试；不直接把 P3 路线图任务批量标记为完成。

## 交付判定

- 阶段二：代码测试通过 + 真实 E2E 通过，里程碑标 `[x]`；有代码但真实依赖未通过，保留 `[~]`；存在明确故障，标 `[!]`。
- 阶段三：TECH-A2A 每个任务以测试和构建结果为准，完成一个小批次同步一个小批次，不提前宣告 v1.0-ga 完成。
