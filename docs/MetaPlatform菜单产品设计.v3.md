# MetaPlatform 菜单产品设计 v3.1（应用中心重构）

> **文档版本**：v3.1
> **核心结构**：**一级菜单 12 项 + 应用中心两层结构 + 三层 AI 入口**
> **设计理念**：**场景驱动 × 角色化分发 × 应用为中心**
> **更新日期**：2026-07-03
> **核心调整**：
> - v3 → v3.1：应用中心改为"以应用为单位"组织
> - v2 → v3：自由页面/门户合并到 Dashbaord，服务编排放到流程中心

---

## 〇、v2 → v3 关键调整

### 0.1 用户决策

| 决策项 | 详情 |
|--------|------|
| ✅ 1. **自由页面** 合并到 **Dashbaord** | 不独立成一级菜单，作为 Dashbaord 下 Tab |
| ✅ 2. **服务编排** 合并到 **流程中心** | 业务流程 / 审批流程 / 服务编排 统一在流程中心 |
| ✅ 3. **门户** 合并到 **Dashbaord** | PC+移动端入口作为 Dashbaord 下 Tab |

### 0.2 v3 vs v2 菜单结构对比

| 模块 | v2 | v3 |
|------|----|----|
| Dashbaord | 4 个 Tab | **6 个 Tab**（+ 门户 + 自由页面） |
| 流程中心 | 7 个 Tab | **9 个 Tab**（+ 业务流程 + 审批流程 + 服务编排） |
| 一级菜单总数 | 12 项 | **12 项（不变）** |

### 0.3 调整理由

1. **保持 12 项一级菜单的认知舒适**
   - 一级菜单 ≤ 12 项是认知舒适区上限
   - 不到 1 年迭代不应轻易增加一级菜单
2. **聚合相似能力**
   - 自由页面是"我的工作空间"的一种能力，归入 Dashbaord 自然
   - 门户是登录入口的另一种形态，归入 Dashbaord 自然
   - 服务编排本质是"流程化的服务调用"，归入流程中心自然
3. **避免一级菜单碎片化**
   - 用户不需要为"门户"和"自由页面"单独记忆
   - 3 类流程（业务/审批/服务编排）统一管理更高效

### 0.4 应用中心重构（v3.1 关键调整）

**用户决策**：应用中心 Tab 应改为"以应用为单位"组织 —— 用户只需创建应用，所有能力（业务数据建模/页面/流程/配置/发布/导出）在应用内完成。

**v3 vs v3.1 应用中心对比**：

| 维度 | v3 | **v3.1** |
|------|----|----|
| 设计理念 | 以"能力类型"分 Tab | **以"应用"为单位组织** |
| Tab 结构 | 6 个 Tab 平铺 | **两层结构（应用列表 + 应用详情 7 个 Tab）** |
| 业务数据建模位置 | 应用中心 Tab 2 | **应用详情 Tab 2** |
| 页面构建位置 | 应用中心 Tab 3 | **应用详情 Tab 3** |
| 流程构建位置 | 应用中心 Tab 4 | **应用详情 Tab 4** |
| 应用配置位置 | 应用中心 Tab 5 | **应用详情 Tab 5** |
| 应用发布位置 | 应用中心 Tab 6 | **应用详情 Tab 6** |
| 应用导出位置 | （新增） | **应用详情 Tab 7** |

**用户路径**：
```
应用中心（默认进入）
    ↓ [看到应用列表]
    ↓ [+ 新建应用] 或 [点击已有应用]
    ↓
应用详情（顶部 Tab 切换）
    ├── 概览（应用首页）
    ├── 业务数据建模（应用内的对象）
    ├── 页面（应用内的页面）
    ├── 流程（应用内的流程）
    ├── 应用配置（权限/OpenAPI/通知/集成）
    ├── 应用发布（多环境/版本/灰度）
    └── 应用导出（CODE EXPORT）
```

**应用详情 7 个 Tab，均 ≤ 7-9，符合认知舒适区**

**为什么合并自由页面到 Dashbaord？**
- 自由页面本质是"用户自定义的可视化页面"
- 与"我的应用""我的数字员工"同属消费侧
- Dashbaord 是千人千面的"我的空间"，自由页面正是"自定义空间"能力的体现

**为什么合并门户到 Dashbaord？**
- 门户是用户登录后的**统一入口**
- 与工作台互补（工作台是数据看板，门户是导航入口）
- 在 Dashbaord 下作为"导航入口 Tab"自然

### 4️⃣ 应用中心（两层结构）

**聚焦**：以**应用为单位**组织 —— 用户只需创建应用，所有能力在应用内

**核心设计理念**：
- ❌ 不是"以能力类型分 Tab"（业务数据建模/页面/流程）
- ✅ 而是"先有应用，应用内 Tab 切换"
- 用户路径：进入应用中心 → 创建应用 → 进入应用详情 → 在 Tab 内做业务数据建模/页面/流程/配置/发布/导出

### 5️⃣ 流程中心（9 个 Tab，**重点优化，BPMN 2.0 规范**）

**聚焦**：业务流程 + 审批流程 + 服务编排

**核心规范**：流程设计器遵循 **BPMN 2.0 规范**（OMG 维护，2011 年发布）

> **BPMN 2.0（Business Process Modeling Notation）** 是由 OMG 维护的流程建模标准图形注解。
> 它提供了一种直观且统一的方式来描述业务流程，支持图形化设计 + 丰富的语义，
> 能表达复杂流程逻辑，适用于业务分析师、开发人员、企业架构师。

**BPMN 2.0 五大基础元素类别**：
1. **流对象（Flow Objects）**：事件、活动、网关
2. **数据（Data）**：数据对象、数据输入、数据输出、数据存储
3. **连接对象（Connecting Objects）**：顺序流、消息流、关联、数据关联
4. **泳道（Swimlanes）**：泳池、泳道
5. **Artifacts**：组、文本注释

  - **审批流程**：单据审批 / 工作流
  - **服务编排流程**：跨系统数据/服务调用

## 九、应用中心核心设计哲学

### 9.1 一句话总结

> **用户只需要"创建应用"，所有能力（业务数据建模/页面/流程/配置/发布/导出）都在应用内完成。**

### 9.2 3 个核心原则

1. **应用为中心**：用户思考的不是"我要建对象/页面/流程"，而是"我要建一个应用（如客户管理）"
2. **应用聚合一切**：一个应用 = 它的对象 + 页面 + 流程 + 配置 + 发布 + 导出
3. **应用可独立运行**：应用可以单独导出、单独部署、单独 OpenAPI

### 9.3 与 v2/v3 的本质差异

| 维度 | v2/v3（以能力为中心） | **v3.1（以应用为中心）** |
|------|----------------------|------------------------|
| 用户思考 | "我要建模/页面/流程" | "我要建一个应用" |
| Tab 切换 | 频繁切换多个一级菜单 | 在应用内 Tab 切换 |
| 能力归属 | 能力独立于应用 | 能力归属于应用 |
| 数据归属 | 对象独立 | 对象归属于应用 |
| 部署粒度 | 平台整体部署 | **应用可独立部署** |
| OpenAPI | 平台统一 API | **应用可独立 OpenAPI** |
| CODE EXPORT | 平台源码 | **应用源码导出** |
