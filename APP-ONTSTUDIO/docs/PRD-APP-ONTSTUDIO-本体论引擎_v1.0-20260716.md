## 1. 模块概述

### 1.1 模块定位

APP-ONTSTUDIO（本体论引擎）是 Mate Platform 的核心建模与管理模块，为平台提供统一语义建模、数据映射、Action 编排与知识图谱可视化的能力。

该模块是平台的「语义中枢」——所有业务模块通过 Ontology 引擎访问数据语义，本体引擎与知识图谱合并为单层，确保全平台数据语义的一致性。

**四大子功能模块：**

| 序号 | 子功能模块 | 核心职责 | 上游技术服务 |
|------|-----------|---------|-------------|
| 1 | 本体论管理 | Concept/Entity/Attribute/Relation/Rule/Action 的可视化建模、版本管理、推理引擎配置 | TECH-ONT, TECH-RULE |
| 2 | 数据中心 | 数据源管理、数据映射（外部数据映射到 Ontology 实体）、数据质量监控、数据血缘 | TECH-DATA, TECH-ONT |
| 3 | Action 编排 | Action 定义（输入/输出/执行逻辑/补偿逻辑）、服务编排、触发规则、执行监控 | TECH-ACTION, TECH-WFE, TECH-RULE |
| 4 | 知识图谱 | 图谱可视化浏览、实体关系探索、图谱查询（Cypher + 可视化查询构建器） | TECH-ONT, TECH-RAG |

**技术依赖：**
- 后端：Java 21 + Spring Boot 3.4 + Spring AI 1.0，Java 包名 `com.metaplatform.ontstudio`
- 前端：React 19 + TypeScript 5.7 + Ant Design 6.0 + AntV X6（图谱可视化）
- 图库：Neo4j 5.x
- 主库：PostgreSQL 17
- API 基础路径：`/api/v1/ontstudio/`

### 1.2 目标用户

| 角色 | 职责描述 | 主要使用场景 |
|------|---------|-------------|
| 本体架构师 | 负责企业本体模型的整体设计与治理 | 创建概念体系、定义关系类型、管理版本与推理规则 |
| 数据工程师 | 负责数据源接入与数据映射 | 配置数据源、创建字段映射、监控数据质量 |
| 业务分析师 | 基于本体模型进行业务分析 | 浏览知识图谱、查询实体关系、查看数据血缘 |
| 系统管理员 | 负责模块配置与权限管理 | 管理用户权限、配置推理引擎、审计操作日志 |
| 应用开发者 | 基于本体模型开发业务应用 | 定义 Action、编排服务、配置触发规则 |

### 1.3 核心价值

1. **统一语义建模**：通过可视化建模工具，让非技术人员也能参与本体模型设计，降低 Ontology 建模门槛
2. **数据到语义的自动映射**：将外部数据源自动映射到 Ontology 实体，消除数据孤岛，实现语义统一
3. **可视化知识探索**：基于 AntV X6 的知识图谱可视化，支持交互式实体关系探索，直观理解业务语义
4. **低代码 Action 编排**：通过拖拽式编排定义服务逻辑，降低集成开发成本
5. **版本化与可追溯**：本体模型全版本管理，支持差异对比与回滚，确保模型变更可追溯

## 3. 功能详情

### 3.1 本体论管理

本体论管理是 APP-ONTSTUDIO 的核心子功能，提供 Concept（概念）、Entity（实体）、Relation（关系）、Rule（规则）的全生命周期管理，以及本体版本管理。

#### 3.1.1 概念管理

##### 3.1.1.1 概念列表页

**入口：** 左侧导航栏 → 点击「本体论管理」→ 默认进入「概念管理」子页签

**页面结构：**
- 顶部：面包屑导航「本体论引擎 / 本体论管理 / 概念管理」
- 工具栏：搜索框、概念类型筛选下拉、状态筛选下拉、「创建概念」按钮、批量操作下拉
- 左侧面板：概念树（树形结构展示概念继承关系）
- 右侧主区域：概念列表表格

**概念列表表格列：**

| 列名 | 说明 | 是否可排序 |
|------|------|-----------|
| 概念名称 | 概念的显示名称 | 是 |
| 概念编码 | 唯一标识符，如 `org.customer` | 是 |
| 描述 | 概念的简要描述 | 否 |
| 属性数量 | 关联的属性总数 | 是 |
| 实例数量 | 关联的实体实例总数 | 是 |
| 父概念 | 继承的父概念名称 | 否 |
| 状态 | 草稿 / 已发布 / 已归档 | 是 |
| 更新时间 | 最后修改时间 | 是 |
| 操作 | 编辑、删除、查看详情 | 否 |

**操作步骤——查看概念列表：**
1. 用户点击左侧导航「本体论管理」→ 系统加载概念管理页面
2. 系统展示概念树（左侧面板）和概念列表表格（右侧主区域）
3. 用户可在搜索框输入关键词 → 点击搜索按钮或按回车 → 表格筛选匹配的概念
4. 用户可点击概念类型下拉 → 选择类型（如「业务概念」「技术概念」）→ 表格按类型筛选
5. 用户可点击列头排序图标 → 表格按该列升序/降序排列
6. 用户可点击左侧概念树中的节点 → 表格筛选显示该概念及其子概念

##### 3.1.1.2 创建概念

**入口：** 概念列表页 → 点击工具栏「创建概念」按钮

**操作步骤：**
1. 用户点击「创建概念」按钮 → 系统弹出「创建概念」对话框（抽屉式，从右侧滑出，宽度 600px）
2. 用户填写表单字段（详见下方字段表）
3. 用户点击「保存」按钮 → 系统执行前端校验
   - 校验通过 → 系统调用后端 API 创建概念 → Toast 提示「概念创建成功」→ 对话框关闭 → 概念列表刷新，新概念出现在列表顶部 → 概念树刷新
   - 校验失败 → 系统在对应字段下方显示红色错误提示，对话框不关闭
4. 用户点击「取消」按钮 → 对话框关闭，不保存数据
5. 用户点击「保存并继续」按钮 → 创建成功后 → Toast 提示「概念创建成功」→ 对话框表单清空，保留对话框打开状态，便于连续创建

**表单字段详情：**

| 字段名 | 类型 | 是否必填 | 校验规则 | 说明 |
|--------|------|---------|---------|------|
| 概念名称 | 文本输入 | 是 | 1-50 字符，不能为空 | 概念的显示名称 |
| 概念编码 | 文本输入 | 是 | 1-100 字符，仅允许小写字母、数字、点号（.），全局唯一 | 格式如 `org.customer`，作为唯一标识 |
| 描述 | 多行文本 | 否 | 最多 500 字符 | 概念的详细说明 |
| 父概念 | 树选择器 | 否 | 必须选择已存在的概念 | 概念继承关系，支持多级选择 |
| 概念类型 | 下拉选择 | 是 | 枚举值：业务概念 / 技术概念 / 枚举概念 | 概念的分类 |
| 图标 | 图标选择器 | 否 | 从预设图标库选择 | 概念在图谱中的展示图标 |
| 颜色 | 颜色选择器 | 否 | 十六进制颜色值 | 概念在图谱中的展示颜色 |
| 标签 | 标签输入 | 否 | 每个标签 1-20 字符，最多 10 个 | 用于分类和检索 |

##### 3.1.1.3 编辑概念

**入口：** 概念列表页 → 点击操作列「编辑」按钮，或点击概念名称进入详情页后点击「编辑」按钮

**操作步骤：**
1. 用户点击操作列「编辑」按钮 → 系统弹出「编辑概念」抽屉对话框，预填充当前概念数据
2. 用户修改需要变更的字段
3. 用户点击「保存」按钮 → 系统执行前端校验
   - 校验通过 → 系统调用后端 API 更新概念 → Toast 提示「概念更新成功」→ 对话框关闭 → 概念列表刷新
   - 校验失败 → 显示对应字段错误提示
4. 若用户修改了概念编码 → 系统弹出二次确认对话框「修改概念编码可能影响已存在的数据映射和 Action 引用，确认修改？」→ 用户点击「确认修改」→ 继续保存；用户点击「取消」→ 返回编辑对话框

##### 3.1.1.4 删除概念

**入口：** 概念列表页 → 点击操作列「删除」按钮

**操作步骤：**
1. 用户点击操作列「删除」按钮 → 系统弹出删除确认对话框
2. 系统显示提示信息「确定删除概念「{概念名称}」吗？此操作不可撤销。」
3. 若该概念存在子概念或关联实体 → 系统在对话框中追加警告「该概念下存在 {N} 个子概念和 {M} 个实体，删除后这些关联将断开，请先处理关联项。」，且「确认删除」按钮置灰
4. 若该概念无关联项 → 「确认删除」按钮可点击
5. 用户点击「确认删除」→ 系统调用后端 API 删除概念 → Toast 提示「概念删除成功」→ 概念列表刷新 → 概念树刷新
6. 用户点击「取消」→ 对话框关闭

##### 3.1.1.5 概念详情页

**入口：** 概念列表页 → 点击概念名称链接

**页面结构：**
- 顶部：概念名称、概念编码、状态标签、操作按钮（编辑、删除、发布、创建实体）
- Tab 页签：基本信息、属性列表、关联实体、关联关系、版本历史
- 基本信息页：展示概念的所有字段信息
- 属性列表页：展示该概念下所有属性的表格
- 关联实体页：展示基于该概念创建的实体实例列表
- 关联关系页：展示该概念参与的所有关系
- 版本历史页：展示该概念的版本变更记录

**操作步骤——查看属性列表：**
1. 用户进入概念详情页 → 点击「属性列表」Tab
2. 系统展示属性表格：属性名称、属性编码、数据类型、是否必填、默认值、描述
3. 用户点击「添加属性」按钮 → 弹出「添加属性」对话框
4. 用户填写属性信息（详见字段表）→ 点击「保存」→ Toast 提示「属性添加成功」→ 属性列表刷新
5. 用户点击属性行的「编辑」按钮 → 弹出编辑对话框 → 修改后点击「保存」→ Toast 提示「属性更新成功」
6. 用户点击属性行的「删除」按钮 → 确认对话框 → 点击「确认删除」→ Toast 提示「属性删除成功」→ 列表刷新

**属性表单字段详情：**

| 字段名 | 类型 | 是否必填 | 校验规则 | 说明 |
|--------|------|---------|---------|------|
| 属性名称 | 文本输入 | 是 | 1-50 字符 | 属性的显示名称 |
| 属性编码 | 文本输入 | 是 | 1-100 字符，仅小写字母+数字+下划线，同一概念内唯一 | 如 `customer_name` |
| 数据类型 | 下拉选择 | 是 | 枚举：String / Integer / Long / Double / Boolean / Date / DateTime / Decimal / JSON / Reference | 属性值的数据类型 |
| 引用概念 | 树选择器 | 条件必填 | 当数据类型为 Reference 时必填 | 引用的目标概念 |
| 是否必填 | 开关 | 否 | 默认关闭 | 实体实例中该属性是否必填 |
| 是否唯一 | 开关 | 否 | 默认关闭 | 该属性值是否全局唯一 |
| 默认值 | 文本输入 | 否 | 需符合数据类型格式 | 新建实体时的默认值 |
| 枚举值 | 动态列表 | 条件必填 | 当数据类型为枚举概念时显示 | 枚举选项的键值对 |
| 描述 | 多行文本 | 否 | 最多 500 字符 | 属性说明 |

##### 3.1.2.2 创建实体

**入口：** 实体列表页 → 点击工具栏「创建实体」按钮

**操作步骤：**
1. 用户点击「创建实体」按钮 → 系统弹出「创建实体」抽屉对话框（宽度 720px）
2. 用户选择所属概念 → 系统根据所选概念动态加载该概念的属性列表，生成属性填写表单
3. 用户填写实体基本信息和属性值
4. 用户点击「保存」按钮 → 系统执行前端校验
   - 校验通过 → 系统调用后端 API 创建实体 → Toast 提示「实体创建成功」→ 对话框关闭 → 实体列表刷新
   - 校验失败 → 显示对应字段错误提示
5. 用户点击「保存并继续」→ 创建成功后 → 对话框表单清空（保留所属概念选择）→ 便于连续创建
6. 用户点击「在图谱中查看」链接（保存成功后出现）→ 跳转到知识图谱页面，定位到新创建的实体节点

**表单字段详情：**

| 字段名 | 类型 | 是否必填 | 校验规则 | 说明 |
|--------|------|---------|---------|------|

| 编排编码 | 文本输入 | 是 | 1-100 字符，仅小写字母+数字+下划线，全局唯一 | 如 `customer_onboarding_flow` |

| 失败策略 | 下拉选择 | 是 | 继续执行 / 中止流程 / 执行补偿 | 节点失败时的整体策略 |

#### 3.4.1 图谱浏览

##### 3.4.1.1 图谱主界面

**入口：** 左侧导航栏 → 点击「知识图谱」Tab

   - 固定节点位置（取消物理引擎的自动布局）

   - 复制节点编码

#### 3.4.2 图谱查询

##### 3.4.2.1 Cypher 查询

**入口：** 图谱主界面 → 点击底部工具栏的「查询」按钮，或顶部搜索栏旁的「高级查询」按钮

**操作步骤：**
1. 用户点击「高级查询」按钮 → 系统弹出查询面板（从底部滑出，高度 400px）
2. 查询面板提供两种模式：
   - **Cypher 模式**：用户直接编写 Cypher 查询语句
   - **可视化构建器模式**：用户通过表单构建查询
3. **Cypher 模式操作：**
   - 用户在代码编辑器中编写 Cypher 语句（如 `MATCH (n:Customer)-[:HAS_ORDER]->(o:Order) WHERE n.name CONTAINS '张' RETURN n, o`）
   - 系统提供语法高亮和自动补全（补全概念名、关系类型、属性名）
   - 用户点击「执行查询」→ 系统在 Neo4j 中执行查询
   - 查询结果展示在画布中：匹配的节点和关系以高亮形式渲染
   - 查询结果同时以表格形式展示在查询面板下方
   - 用户可点击「保存查询」→ 输入查询名称 → 保存为常用查询
4. **可视化构建器模式操作：**
   - 用户点击「添加节点条件」→ 选择概念 → 配置属性过滤条件
   - 用户点击「添加关系条件」→ 选择关系类型 → 配置关系方向和跳数
   - 用户点击「添加返回项」→ 选择返回的属性
   - 系统自动生成 Cypher 语句预览
   - 用户点击「执行查询」→ 执行并展示结果
5. 用户点击「清空」→ 清除查询条件和结果
6. 用户点击「关闭面板」→ 收起查询面板

##### 3.4.2.2 常用查询管理

**入口：** 查询面板 → 点击「常用查询」按钮

**操作步骤：**
1. 用户点击「常用查询」按钮 → 系统展示已保存的查询列表
2. 用户点击某个查询 → 加载查询语句到编辑器
3. 用户点击查询的「编辑」按钮 → 修改查询名称或描述
4. 用户点击查询的「删除」按钮 → 确认后删除
5. 用户点击「设为默认」→ 该查询设为图谱默认加载查询

#### 3.4.3 图谱布局切换

##### 3.4.3.1 布局切换

**入口：** 图谱主界面 → 底部工具栏布局切换下拉

**操作步骤：**
1. 用户点击底部工具栏的布局下拉按钮 → 系统展示布局选项列表
2. 可选布局类型：
   - **力导向布局**（默认）：节点根据斥力和引力自动排列，适合探索性浏览
   - **层次布局**：节点按层级排列，从上到下展示继承和依赖关系
   - **环形布局**：节点排列成环形，适合展示同级实体
   - **网格布局**：节点排列成网格，适合大量节点的整齐展示
   - **Dagre 层次布局**：基于 DAG 的层次布局，适合展示有向无环关系
3. 用户选择布局类型 → 系统以动画形式平滑过渡到新布局（过渡时长 500ms）
4. 布局切换后，用户已固定的节点位置保持不变

##### 3.4.3.2 布局参数调整

**入口：** 布局切换下拉旁的「布局设置」齿轮按钮

**操作步骤：**
1. 用户点击「布局设置」按钮 → 系统弹出布局参数面板
2. 根据当前布局类型，系统展示不同的参数：
   - **力导向布局**：斥力强度、引力强度、链接距离、动画速度
   - **层次布局**：层级方向（上→下/左→右）、层级间距、节点间距
   - **环形布局**：半径、起始角度、方向（顺时针/逆时针）
   - **网格布局**：列数、行间距、列间距
3. 用户调整参数 → 画布实时预览效果
4. 用户点击「应用」→ 保存当前布局参数
5. 用户点击「恢复默认」→ 参数重置为默认值

##### 3.4.3.3 图谱导出

**入口：** 图谱主界面 → 底部工具栏「导出」按钮

**操作步骤：**
1. 用户点击「导出」按钮 → 系统弹出导出选项面板
2. 用户选择导出格式：
   - **PNG 图片**：导出当前画布截图
   - **SVG 矢量图**：导出矢量图形
   - **JSON 数据**：导出当前图谱的节点和关系数据（JSON 格式）
   - **GraphML**：导出为 GraphML 格式（可被其他图谱工具导入）
3. 用户选择导出范围：当前视图 / 全部节点 / 选中节点
4. 用户点击「确认导出」→ 系统生成文件 → 下载到本地 → Toast 提示「导出成功」

### 7.3 Neo4j 图模型

```
节点标签 (Labels):
  (:Concept {id, name, code, type, color})
  (:Entity {id, name, code, conceptId, source, status})
  (:DataSource {id, name, type, status})

关系类型 (Relationship Types):
  [:HAS_ATTRIBUTE {required, unique, defaultValue}]
  [:INSTANCE_OF]                    // Entity -> Concept
  [:HAS_RELATION {relationTypeId, validFrom, validTo}]
  [:MAPPED_FROM {mappingId, syncTime}]
  [:TRIGGERS {triggerId}]
  [:EXECUTES {executionId}]
  [:DERIVED_FROM {dataSourceId}]
```

### 7.4 关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 概念继承 | 树形结构（parent_id） | 支持概念的多级继承，树结构简单且满足业务需求 |
| 属性存储 | 纵表设计（entity_value 表） | 支持动态属性扩展，无需修改表结构 |
| 关系存储 | PostgreSQL（结构化） + Neo4j（图谱）双写 | PostgreSQL 保证事务一致性，Neo4j 支持高效图谱查询 |
| 规则存储 | JSON 字段存储条件和动作 | 规则结构灵活多变，JSON 支持动态结构 |
| 编排拓扑 | JSON 存储 | 拓扑结构灵活，JSON 便于前端渲染和后端解析 |
| 版本管理 | 快照 + 变更日志 | 快照支持快速回滚，变更日志支持差异对比 |
| 数据同步 | 异步任务 + Outbox 模式 | 遵循平台 Outbox 模式约束，防止数据丢失 |
| 图谱缓存 | Neo4j 作为图谱缓存层 | 避免每次从 PostgreSQL 构建图谱，提升查询性能 |
