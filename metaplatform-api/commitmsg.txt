feat(publish): rollback, runtime health banner, FAB unification

- GET /api/runtime/health exposes the platform-wide docker daemon
  status (ok | degraded + error). Used by the publish tab's new
  RuntimeHealthBanner to surface "运行时未隔离（Docker 不可达）" at
  the very top of /apps/:id/publish so users see the degraded state
  before they touch anything.

- POST /api/apps/:id/restore { publicationId } rolls the live
  runtime back to an archived publication without restarting any
  container. Internally it swaps the BASE_TO_LATEST alias to point
  at the archived slug and updates applications.runtime_alias_slug
  so the alias persists across platform restarts. The archived
  container is left running, /app/<historicalSlug> keeps resolving.

- AppPublish's "已发布环境" list now has a "切换为主版本" button on
  each archived row. Clicking it triggers the restore endpoint
  with the publication id, then refreshes the list so the row
  ordering swaps (the restored row is now production, the old live
  becomes archived).

- PageAIAssistant: the FAB button is now the same DOM in closed and
  open states; the chat panel grows above the button via
  flex-col-reverse, so closing the panel returns the user to a
  visible FAB without remounting. Minimize merged with Close
  (both call setIsOpen(false) and return to FAB).

- metaplatform-runtime:v1 is now the published image tag the
  orchestrator expects. A docker build + smoke test run through
  scripts/test-alias.py confirmed the image builds, the health
  check returns healthy, and the published app serves real data
  over the reverse-proxy.

Tested end-to-end with scripts/test-restore.py:

  v2: historical=...-thtd1j port=31046
  live before restore = 31046
