## 1. 模块概述

### 1.1 模块定位

APP-DASHBOARD 是 Mate Platform 的全局统一入口与数据看板。用户登录平台后看到的第一个页面即为仪表盘首页。仪表盘聚合来自全平台各业务模块的关键信息，实现"一屏掌控全局"的运营体验。

仪表盘不承载复杂业务操作，其核心职责是：

- 作为全平台的统一入口，引导用户进入各功能中心
- 聚合展示用户关注的工作待办、审批任务、数字员工状态
- 可视化呈现企业运营关键指标
- 统一接收和管理全平台的消息通知
- 提供个人偏好设置与权限查看入口

### 1.2 目标用户

| 角色 | 使用场景 | 关注重点 |
|---|---|---|
| 平台管理员 | 监控平台整体运行状态 | 系统资源使用率、活跃用户数、在线服务数、异常告警 |
| 业务运营人员 | 处理日常待办与审批 | 待办事项列表、审批任务、业务指标趋势 |
| 数字员工管理者 | 监控数字员工运行情况 | 在线数字员工数、任务执行状态、执行成功率 |
| 普通用户 | 日常工作台 | 个人待办、快捷入口、消息通知 |

### 1.3 核心价值

- **统一入口**：所有用户通过仪表盘进入平台，根据角色展示差异化内容
- **实时感知**：聚合关键业务指标、待办事项、系统通知，实时反映平台运行状态
- **个性化定制**：用户可自定义看板布局、快捷入口、关注指标
- **运营提效**：将高频操作前置，减少用户在多模块间的跳转成本

**欢迎信息栏：**
- 显示用户姓名 + 问候语（根据当前时间：上午好/下午好/晚上好）
- 显示当前日期与星期
- 显示今日概要数据：今日待办数、今日已处理数、今日通知数

**交互：**

1. 页面加载时 → 系统调用 `GET /api/v1/dashboard/summary` 获取今日概要数据
2. 欢迎信息栏淡入显示
3. 下方各卡片区域并行加载数据，每个卡片区域显示骨架屏（Skeleton）占位
4. 数据加载完成后，骨架屏替换为实际内容，卡片渐入显示
5. 若某个卡片数据加载失败 → 卡片区域显示"数据加载失败，点击重试"文案 + 重试按钮

**反馈：**
- 首屏渲染 < 1.5 秒
- 各卡片数据并行加载，互不阻塞
- 加载失败时提供重试入口

#### 3.1.2 待办事项卡片

**入口：** 首页工作台区域，左上角第一个卡片

**卡片内容：**

| 元素 | 说明 |
|---|---|
| 卡片标题 | "待办事项"，右侧显示待办总数红色角标（如 8） |
| 分类标签 | 审批待办(5) / 任务待办(2) / 工单待办(1) |
| 待办列表 | 显示前 5 条待办，按优先级和创建时间排序 |
| 查看全部 | 卡片底部「查看全部」链接 |

**待办列表项内容：**

| 元素 | 说明 |
|---|---|
| 优先级标识 | 左侧色条：紧急(红) / 高(橙) / 中(黄) / 低(灰) |
| 待办标题 | 单行文本，超出省略号 |
| 来源模块 | 灰色小字标签，如"采购审批""任务中心" |
| 截止时间 | 灰色小字，如"今天 18:00 截止"，已逾期显示红色 |
| 来源图标 | 左侧小图标，区分审批/任务/工单 |

**操作步骤：**

1. 用户进入首页 → 卡片区域显示骨架屏 → 数据加载完成后展示待办列表
2. 用户点击分类标签（如"审批待办"）→ 列表筛选为对应类型的待办 → 角标数字更新
3. 用户点击某条待办项 → 跳转到对应业务模块的处理页面：
   - 审批待办 → 跳转 `APP-APPHUB` 审批详情页 `/apphub/approvals/{approvalId}`
   - 任务待办 → 跳转 `APP-DW` 任务详情页 `/dw/tasks/{taskId}`
   - 工单待办 → 跳转对应工单系统
4. 用户点击「查看全部」→ 展开全屏待办抽屉（Drawer），显示完整待办列表
5. 在全屏抽屉中：
   - 顶部筛选区：支持下拉筛选「待办类型」「来源模块」「优先级」
   - 列表分页显示，每页 20 条
   - 点击某条 → 同步骤 3 跳转处理
6. 用户处理完待办返回首页 → 待办列表自动刷新 → 已处理的待办从列表移除 → 角标数字减 1

**交互详情：**
- 待办列表按优先级排序：紧急 > 高 > 中 > 低，同优先级按创建时间倒序
- 截止时间已逾期的待办项标题变红，并在标题前显示红色感叹号图标
- 鼠标悬停待办项 → 背景色变为浅灰高亮
- WebSocket 实时推送：新待办到达时 → 卡片顶部滑入新项 + 角标数字增加 + 顶部铃铛图标闪烁

**反馈：**
- 待办列表加载 P95 < 300ms
- 新待办推送延迟 < 1 秒
- 点击待办项跳转响应 < 200ms

#### 3.1.3 审批任务列表

**入口：** 首页工作台区域，待办事项卡片右侧的「审批任务」卡片，或待办卡片中点击"审批待办"分类

**卡片内容：**

| 元素 | 说明 |
|---|---|
| 卡片标题 | "审批任务"，右侧显示待审批总数 |
| 快捷操作 | 卡片右上角「批量审批」按钮（灰色，3 条以上可用） |
| 审批列表 | 显示前 5 条待审批任务 |
| 查看全部 | 卡片底部「查看全部」链接 |

**审批列表项内容：**

| 元素 | 说明 |
|---|---|
| 申请人头像 | 左侧圆形头像 32x32 |
| 申请人姓名 | 加粗 14px |
| 申请标题 | 如"采购申请单 - 办公设备采购" |
| 申请金额 | 如"￥12,500"（如适用） |
| 申请时间 | 灰色小字，如"2026-07-16 10:30" |
| 审批状态 | 灰色标签"待审批" |
| 快捷按钮 | 列表项右侧两个按钮：「通过」(绿色)、「拒绝」(红色) |

**操作步骤 - 查看审批详情：**

1. 用户在审批任务卡片中看到待审批列表
2. 用户点击审批列表项（非按钮区域）→ 跳转到 `APP-APPHUB` 审批详情页 `/apphub/approvals/{approvalId}`
3. 在审批详情页查看完整表单数据和审批流程图
4. 在详情页内完成审批操作（通过/拒绝/转交/退回）
5. 审批完成 → 返回仪表盘 → 审批列表自动刷新 → 已处理的审批从列表移除

**操作步骤 - 快捷审批（通过）：**

1. 用户在审批列表项右侧点击「通过」按钮
2. 系统弹出审批确认弹窗（Modal），内容：
   - 标题："确认通过审批"
   - 审批信息摘要：申请人、申请标题、金额
   - 审批意见输入框（多行文本，非必填，placeholder="请输入审批意见"）
   - 「取消」灰色按钮 / 「确认通过」绿色主按钮
3. 用户输入审批意见（可选）→ 点击「确认通过」
4. 按钮显示 loading 状态 → 调用 `PUT /api/v1/dashboard/todos/{todoId}/handle`
5. 审批成功 → 关闭弹窗 + Toast 绿色提示"审批通过" + 列表移除该项 + 角标数字减 1
6. 审批失败 → Toast 红色提示"操作失败，请重试" + 弹窗保持打开

**操作步骤 - 快捷审批（拒绝）：**

1. 用户在审批列表项右侧点击「拒绝」按钮
2. 系统弹出拒绝确认弹窗（Modal），内容：
   - 标题："确认拒绝审批"
   - 审批信息摘要
   - 拒绝原因输入框（多行文本，必填，placeholder="请输入拒绝原因"）
   - 「取消」灰色按钮 / 「确认拒绝」红色主按钮
3. 用户输入拒绝原因 → 点击「确认拒绝」
4. 校验：拒绝原因非空 → 失败则输入框下方红字提示"请输入拒绝原因"
5. 校验通过 → 按钮 loading → 调用 API → 关闭弹窗 + Toast"已拒绝" + 列表刷新

**操作步骤 - 批量审批：**

1. 用户点击卡片右上角「批量审批」按钮
2. 审批列表进入多选模式：每条左侧出现复选框
3. 用户勾选多条审批任务
4. 底部出现操作栏：「已选 N 项」+「批量通过」绿色按钮 +「批量拒绝」红色按钮 +「取消」按钮
5. 用户点击「批量通过」→ 弹出确认弹窗 → 输入审批意见 → 确认 → 批量处理
6. 批量处理结果：成功数 / 失败数 → Toast 提示"批量审批完成：成功 N 条，失败 M 条"
7. 用户点击「取消」→ 退出多选模式

**交互详情：**
- 审批列表按申请时间倒序排列
- 快捷审批仅支持"通过"和"拒绝"两种操作，转交/退回需进入详情页
- 列表项 hover 时背景色高亮，快捷按钮区域不遮挡点击
- WebSocket 推送：新审批到达时，列表顶部滑入新项

**反馈：**
- 快捷审批操作响应 P95 < 500ms
- 批量审批最多支持 50 条
- 审批完成后列表实时刷新，无需手动刷新页面

|---|---|

| 状态摘要 | 今日执行任务数 / 成功率 / 平均执行时长 |
| 数字员工列表 | 显示前 5 个数字员工的实时状态 |
| 查看全部 | 卡片底部「前往数字员工中心」链接 |

**数字员工列表项内容：**

| 元素 | 说明 |
|---|---|
| 员工头像 | 左侧圆形头像 32x32 |
| 员工名称 | 加粗 14px，如"财务助手" |
| 当前状态 | 状态标签：运行中(绿色脉冲动画) / 空闲(灰色) / 离线(浅灰) / 异常(红色) |
| 当前任务 | 灰色小字，如"正在处理：发票识别 #20260716-001" |
| 今日统计 | 灰色小字，如"今日完成 12 个任务" |
| 查看详情 | 右侧「详情」链接按钮 |

**操作步骤：**

1. 用户进入首页 → 卡片加载在线数字员工状态
2. 用户查看数字员工列表，了解各数字员工的运行状态
3. 状态为"运行中"的数字员工 → 头像旁显示绿色脉冲动画 → 当前任务信息实时更新
4. 状态为"异常"的数字员工 → 头像旁显示红色感叹号 → 点击感叹号弹出异常详情浮层（Popover），显示异常原因和发生时间
5. 用户点击某数字员工「详情」链接 → 跳转到 `APP-DW` 数字员工详情页 `/dw/workers/{workerId}`
6. 在数字员工中心查看完整执行记录、对话历史、任务统计
7. 用户点击「前往数字员工中心」→ 跳转到 `APP-DW` 首页 `/dw`

**交互详情：**
- 卡片顶部状态摘要区域展示三个关键数字：今日执行任务数 / 成功率 / 平均执行时长
- 数字以计数动画方式从 0 滚动到实际值（500ms 动画）
- 状态为"运行中"的数字员工排在前方，其次"空闲"，最后"离线"
- WebSocket 推送：数字员工状态变更时（如开始执行任务/完成任务/异常）实时更新卡片
- 数字员工异常时，卡片右上角显示红色告警角标

**反馈：**
- 状态数据刷新间隔 30 秒
- WebSocket 状态变更推送延迟 < 1 秒
- 异常告警在顶部导航栏铃铛通知中同步推送

#### 3.4.3 已读未读管理

**入口：** 通知中心页面，通知列表中的操作

**操作步骤 - 单条标记已读：**

1. 用户在通知列表中，鼠标悬停某条未读通知 → 右侧显示操作按钮

3. 通知标题字重从加粗变为常规 → 左侧蓝色小圆点消失
4. 铃铛角标未读数减 1

**操作步骤 - 单条标记未读：**

1. 用户在通知列表中，鼠标悬停某条已读通知 → 右侧显示操作按钮
2. 用户点击「标记未读」按钮 → 该通知从已读变为未读
3. 铃铛角标未读数加 1

**操作步骤 - 全部标记已读：**

1. 用户点击通知列表上方「全部标记已读」按钮
2. 系统调用 `PUT /api/v1/dashboard/notifications/read-all`
3. 所有未读通知标记为已读 → 铃铛角标数字归零 → Toast"全部标记为已读"

**操作步骤 - 批量删除：**

1. 用户在通知列表中勾选多条通知（每条左侧出现复选框）
2. 底部出现操作栏：「已选 N 项」+「批量删除」按钮 +「取消」按钮
3. 用户点击「批量删除」→ 弹出确认弹窗"确定删除选中的 N 条通知？"
4. 用户点击「确认删除」→ 批量删除 → Toast"已删除 N 条通知"
5. 用户点击「取消」→ 退出多选模式

**操作步骤 - 清空全部通知：**

1. 用户在通知列表上方点击「清空全部」按钮
2. 弹出确认弹窗，红色警告图标 + 文案"确定清空所有通知？此操作不可恢复。"
3. 用户点击「确认清空」→ 清空所有通知 → Toast"已清空所有通知"
4. 通知列表显示空状态

**交互详情：**
- 未读通知视觉标识：标题加粗 + 左侧蓝色小圆点
- 已读通知视觉标识：标题常规字重 + 无小圆点
- 通知自动标记已读：用户点击通知跳转后自动标记
- 删除操作有二次确认，防止误删

**反馈：**
- 标记已读/未读响应 < 300ms
- 全部标记已读响应 < 500ms
- 批量删除响应 < 1 秒

### 5.2 下游消费

| 下游方 | 消费内容 | 说明 |
|---|---|---|
| 所有 APP 模块 | 仪表盘作为统一入口，导航至各模块 | 用户提供导航跳转 |
| 所有 APP 模块 | 各模块通过 TECH-MSG 向仪表盘推送通知和待办 | 通知/待办数据来源 |
| TECH-OBS | 用户行为审计日志 | 仪表盘记录用户操作行为 |

### 5.3 交互流程

```
用户登录 → TECH-IAM 认证 → 返回 JWT Token
  ↓
前端加载仪表盘 → 并行调用工作台/指标/待办/通知 API
  ↓
TECH-GW 路由请求 → 各后端服务返回数据
  ↓
TECH-MSG 推送实时通知 → WebSocket 推送到前端
  ↓
TECH-OBS 记录用户行为审计日志
  ↓
用户点击待办/快捷入口 → 跳转到对应业务模块
  ↓
业务模块处理完成 → 通过 TECH-MSG 推送状态更新 → 仪表盘实时刷新
```

### 5.4 数据流转

```
各业务模块产生事件（审批创建、任务完成、告警触发等）
  ↓
TECH-MSG (Kafka/RabbitMQ) 消息队列
  ↓
仪表盘消息消费 → 通知/待办写入 PostgreSQL
  ↓
WebSocket 推送到前端 → 用户看到实时更新
  ↓
指标数据从 TECH-OBS / TECH-WFE / TECH-AGENT 获取 → 聚合后展示在指标看板
  ↓
SuperAI / APP-DW 生成交付材料 → 通过事件通知仪表盘
  ↓
仪表盘写入 Deliverable 记录 → 推送通知 → 用户在历史交付材料中查看
```

├── sortOrder: Integer
├── createdAt: Timestamp
└── updatedAt: Timestamp

├── status: Enum                 // [ACTIVE, INACTIVE]
├── createdAt: Timestamp
